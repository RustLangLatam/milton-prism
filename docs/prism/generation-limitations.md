# Generation Limitations

Known limitations of the autonomous generation agent. These are tracked here rather than as code TODOs because the fix belongs in the generation layer, not in the service or push layer.

---

## Shared-file conflict: each service regenerates shared monorepo files independently

### What happens

The generation agent processes each service in isolation. When two or more services in the same migration include a shared monorepo file (e.g. `pkg/gateway/common/error/message_error.go`), the agent generates that file once per service, each version tailored to only that service's domain. None of the resulting versions is canonical: each includes only its own service's error codes or type references, and lacks those of the other services.

Observed in migration 10018: `pkg/gateway/common/error/message_error.go` was generated by `articles`, `profile`, and `user`. The `articles` version added `articlesErrorMessages` to the gateway lookup; the `profile` version added `profileErrorMessages`; the `user` version added `userErrorMessages`. No single version included all three.

### Current mitigation

`PublishMigration` (MIG211) detects divergent content on the same path before writing anything and returns `Failure_Artifact_Path_Conflict`, listing the conflicting path and services. The migration stays in READY and is retryable once the generation issue is resolved. No silently incomplete files are pushed.

### Root cause

The agent sees one service boundary specification at a time. It has no view of the other services being generated in the same migration, so it cannot produce a merged, complete version of shared files.

This is the same class of problem as cross-service FK variance (where a foreign-key type generated for service A diverges from what service B expects): **agent variance from lack of global context**.

### Correct fix (not implemented — generation layer task)

One of two approaches:

1. **Post-generation merge pass**: after all services are generated, a merge step identifies files that appear in more than one service output and produces a canonical merged version (e.g. by unioning the service-specific additions).

2. **Shared-file generation phase**: add an explicit generation phase for files that are monorepo-wide (gateway error maps, shared proto types, etc.) that runs once after all service protos are finalized and has full visibility of all services.

Both approaches require changes to the generation worker pipeline, not to the push or migration service. This task should be paired with the FK cross-service variance fix.

### Tracking

- Blocked: `PublishMigration` on any migration with shared-file conflicts until re-generation produces non-conflicting artifacts.
- See: `domain.ErrCodeArtifactConflict` (MIG211) for the push-layer guard.
- Related: cross-service FK type variance (generation variance, same root cause, separate manifestation).
