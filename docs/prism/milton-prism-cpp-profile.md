# Milton Prism — C++ (C++20 / grpc++ / Drogon) Language Profile

Peer document to `milton-prism-go-profile.md`, `milton-prism-python-profile.md`, `milton-prism-node-profile.md`, `milton-prism-rust-profile.md`, `milton-prism-java-profile.md`, `milton-prism-ruby-profile.md` and `milton-prism-csharp-profile.md`. This is the canonical **C++20 + CMake + grpc++ / Drogon** mechanism reference for all C++ services generated by Milton Prism. It composes with the **Architecture Canon** (`milton-prism-architecture-canon.md`) exactly as the other Profiles do: the Canon's principles, the proto/AIP contract, the error taxonomy, the dependency rule, and the testing philosophy all still apply unchanged. This document fills only the C++ *mechanisms*.

> **Scope note:** this Profile defines the *output* language for generated microservices. It is independent of the analysis engine's input languages. Supported stores are **MongoDB** (mongocxx/bsoncxx, §A.5–A.6) **and SQL — PostgreSQL / MySQL/MariaDB — via artisanal parametrized SQL (§A.13)**; transports are **gRPC** (grpc++) **and HTTP** (Drogon). Branch on the boundary spec's `store:` field: `store: mongodb` → the mongocxx collections/repos; `store: postgres` OR `store: mysql` → §A.13's hand-written parametrized-SQL repos. **NATS** remains a hole — if requested, stop and write a blocker note rather than improvising.

> **No reference implementation in the repo.** Unlike the Python profile (which ships a `python/` skeleton), the C++ deliverable is generated **entirely from this document plus the proto contract**. There is no pre-existing `cpp/` tree to read. Follow the patterns in this document literally; they ARE the reference.

---

## A.1 Stack (fixed)

- **Language / runtime:** **C++20**, compiled with **g++** (the agent image ships g++/cmake/ninja, glibc). The whole workspace must pass `cmake --build` with zero errors and the suite must pass via `ctest`.
- **Dependency manager / build:** **CMake (>=3.20) + Ninja**. ONE top-level `CMakeLists.txt` at the source root `cpp/` plus a per-service `CMakeLists.txt` under `cpp/services/<service>/`. **Build gate: `cmake -S . -B build -G Ninja && cmake --build build` (the BUILD GATE) + `ctest`.**
- **Dependencies via `pkg-config` (proven Debian path) against PREINSTALLED libraries (CRITICAL).** The agent image (Debian trixie) ships grpc++, protobuf, mongocxx, bsoncxx and libpqxx **preinstalled via apt**. Debian ships pkg-config `.pc` files for them but does NOT always ship CMake config packages, so the **certified, known-good mechanism is `pkg_check_modules` (pkg-config)** — see the golden CMakeLists template in §A.1.1 below, which is exactly what the image's pre-warm guard compiles + links. `find_package(... CONFIG)` MAY also work where a config package exists, but **pkg-config is the proven Debian path**. Either way the generated CMake **MUST resolve every third-party library against those preinstalled libs** — NEVER with CMake `FetchContent`/`ExternalProject` to download grpc/protobuf/mongocxx. Re-downloading and recompiling those libraries blows the container timeout and fails the gate. Keep the dependency set minimal. **Never** ship `build/`, `CMakeFiles/`, `*.o`, `*.a`, `*.so`, or `CMakeCache.txt` in the deliverable (see A.10 Cleanliness).
- **Dependency hygiene:** the CMake links ONLY the preinstalled libraries the code actually uses — no unused/leftover `pkg_check_modules`/`target_link_libraries` entry; keep the set minimal.
- **gRPC:** **grpc++** — the host is built with `grpc::ServerBuilder` producing a `grpc::Server` (HTTP/2). The service class **subclasses the generated `<Foo>::Service` base** from `<svc>.grpc.pb.h`. The message + service stubs are generated at build time by **`protoc`** (`--cpp_out` for messages → `*.pb.cc/.pb.h`) plus the **`grpc_cpp_plugin`** (`--grpc_out` → `*.grpc.pb.cc/.grpc.pb.h`) driven by a CMake custom command.
- **HTTP:** **Drogon** — `drogon::HttpController` registered on `drogon::app().run()`, the only entrypoint. (See A.4-HTTP and the HTTP generation prompt.)
- **Proto codegen:** a CMake custom command runs `protoc` against the canonical `protobuf/proto` include root: `--cpp_out` (messages → `*.pb.cc/.pb.h`) and `--grpc_out` with `--plugin=protoc-gen-grpc=$(which grpc_cpp_plugin)` (service → `*.grpc.pb.cc/.grpc.pb.h`); generated C++ lands in the **CMake build dir** and is **NEVER edited by hand** and **NEVER shipped**.
- **Mongo driver:** **mongocxx / bsoncxx** (pkg-config tokens `libmongocxx`/`libbsoncxx` — the include dirs require BOTH `${MONGOCXX_INCLUDE_DIRS}` AND `${BSONCXX_INCLUDE_DIRS}`) — `mongocxx::collection` documents, repositories on top, mapping bson ⇄ domain by hand. `mongocxx::instance` must be created exactly once in `main()`.
- **SQL:** **artisanal parametrized SQL** — `libpqxx` (`pqxx::connection`, `exec_params`) for PostgreSQL, `mysql-connector-c++` for MySQL/MariaDB — hand-written repositories, NO ORM (§A.13).
- **Config:** environment variables (`MONGO_URI`, `MONGO_DATABASE`, the SQL connection string, `GRPC_HOST`, `GRPC_PORT`, `JWT_SECRET`) read via `std::getenv`. Never hardcode secrets.
- **IDs:** Mongo → BSON `int64` (`std::int64_t`) for the proto `uint64 identifier` field, never narrowed. SQL → autoincrement/identity primary key (`SERIAL`/`BIGINT AUTO_INCREMENT`).

### A.1.1 Pinned dependency set (must match the preinstalled apt libraries)

The generated CMake MUST resolve exactly these preinstalled libraries (kept in lockstep with the agent-image author) and **NEVER** FetchContent anything. The certified resolution on the Debian-trixie agent image is **pkg-config** (`pkg_check_modules`); `find_package(... CONFIG)` may also work where a config package exists, but pkg-config is the proven path:

| Purpose | Library (apt package) | Resolution (pkg-config token — certified) |
|---------|------------------------|-------------------------------------------|
| gRPC server | grpc++ (`libgrpc++-dev`) | `pkg_check_modules(GRPCPP REQUIRED grpc++)` |
| gRPC codegen plugin | `grpc_cpp_plugin` (`protobuf-compiler-grpc`) | `$(which grpc_cpp_plugin)` in the protoc custom command (system, ABI-matched) |
| Protobuf runtime + codegen | protobuf (`libprotobuf-dev` + `protobuf-compiler`) | `pkg_check_modules(PROTOBUF REQUIRED protobuf)` |
| Mongo store | mongocxx/bsoncxx (`libmongocxx-dev`/`libbsoncxx-dev`) | `pkg_check_modules(MONGOCXX REQUIRED libmongocxx)` + `pkg_check_modules(BSONCXX REQUIRED libbsoncxx)` |
| SQL store (PostgreSQL) | libpqxx (`libpqxx-dev`) | `pkg_check_modules(PQXX REQUIRED libpqxx)` |
| SQL store (MySQL/MariaDB) | mysql-connector-c++ (`libmysqlcppconn-dev`) | pkg-config (mysqlcppconn) |
| HTTP transport | Drogon | `find_package(Drogon CONFIG REQUIRED)` (or pkg-config) |
| Auth (JWT) | jwt-cpp (header-only, vendored at `/usr/local/include`) | `#include <jwt-cpp/jwt.h>` — NO find_package; backed by OpenSSL → link `ssl crypto` |
| Tests | GoogleTest (`libgtest-dev` if available) | `find_package(GTest)` — else a minimal assert-based test main |

> **Use ONLY these preinstalled libraries; NEVER FetchContent.** If a needed library is missing from the image, stop and report an environment blocker rather than reaching to the network. Use the Mongo libs for `store: mongodb`, the SQL libs for `store: postgres|mysql` — never both store stacks at once. jwt-cpp is header-only and vendored at `/usr/local/include` (no find_package, no link target beyond OpenSSL's `ssl crypto`); GoogleTest is optional — fall back to a minimal assert-based test main when absent. `protoc` + `grpc_cpp_plugin` come from the SYSTEM Debian `protobuf-compiler` / `protobuf-compiler-grpc` packages, ABI-matched to `libgrpc++-dev`/`libprotobuf-dev` — resolve the plugin path with `which grpc_cpp_plugin`.

#### A.1.1.1 Golden CMakeLists template (proven, known-good — use VERBATIM)

This is exactly the mechanism the agent image's pre-warm guard proves at build time (compiles the generated grpc stubs + links grpc++/mongocxx/jwt-cpp). Use it verbatim as the per-service `CMakeLists.txt` shape, substituting `<svc>`/sources. For a SQL store, swap the mongocxx/bsoncxx blocks for `pkg_check_modules(PQXX REQUIRED libpqxx)` and link `${PQXX_LIBRARIES}`:

```cmake
cmake_minimum_required(VERSION 3.20)
project(<svc> CXX)
set(CMAKE_CXX_STANDARD 20)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

find_package(PkgConfig REQUIRED)
pkg_check_modules(GRPCPP REQUIRED grpc++)
pkg_check_modules(PROTOBUF REQUIRED protobuf)
pkg_check_modules(MONGOCXX REQUIRED libmongocxx)
pkg_check_modules(BSONCXX REQUIRED libbsoncxx)

# protoc codegen: add a custom_command running
#   protoc -I protobuf/proto --cpp_out=${CMAKE_CURRENT_BINARY_DIR} \
#          --grpc_out=${CMAKE_CURRENT_BINARY_DIR} \
#          --plugin=protoc-gen-grpc=$(which grpc_cpp_plugin) <svc>.proto
# emitting <svc>.pb.cc/.pb.h + <svc>.grpc.pb.cc/.grpc.pb.h into the build dir (NEVER shipped).

add_executable(<svc> main.cpp ... <svc>.pb.cc <svc>.grpc.pb.cc)
target_include_directories(<svc> PRIVATE
    ${CMAKE_CURRENT_BINARY_DIR}
    ${GRPCPP_INCLUDE_DIRS} ${PROTOBUF_INCLUDE_DIRS}
    ${MONGOCXX_INCLUDE_DIRS} ${BSONCXX_INCLUDE_DIRS})   # mongocxx needs BOTH include dirs
target_link_directories(<svc> PRIVATE
    ${GRPCPP_LIBRARY_DIRS} ${PROTOBUF_LIBRARY_DIRS}
    ${MONGOCXX_LIBRARY_DIRS} ${BSONCXX_LIBRARY_DIRS})
target_link_libraries(<svc> PRIVATE
    ${GRPCPP_LIBRARIES} ${PROTOBUF_LIBRARIES}
    ${MONGOCXX_LIBRARIES} ${BSONCXX_LIBRARIES}
    ssl crypto)                                          # jwt-cpp (header-only) → OpenSSL
```

(For an **HTTP/Drogon** service, drop the `--grpc_out` half of the protoc command and the `.grpc.pb.cc` source — message classes only — and link Drogon instead of the grpc++ service base; see the HTTP prompt. For a **SQL** service, drop the mongocxx/bsoncxx blocks entirely and use the libpqxx pkg-config block.)

## A.2 Repository layout (mirrors the Go/Python/Node/Rust/Java/Ruby/C# monorepo conceptually)

The agent writes everything under the source root **`cpp/`** (the assembler renames `cpp/` → `core/` in the final deliverable, exactly as Node renames `node/` → `core/`, Java renames `java/` → `core/`, Ruby renames `ruby/` → `core/` and C# renames `csharp/` → `core/`). Protos and docs are siblings of the source root.

```
cpp/                                            # source root (→ core/ in deliverable)
├── CMakeLists.txt                              # top-level: find_package the preinstalled libs, add subdirs
└── services/
    └── <service>/
        ├── CMakeLists.txt                      # per-service target + protoc/grpc_cpp_plugin custom command
        ├── main.cpp                            # entrypoint: build grpc::ServerBuilder & start grpc::Server
        ├── domain/                             # framework-free structs/aliases + typed errors
        │   ├── model/                          # structs/type-aliases of the generated *.pb.h messages
        │   └── error/                          # DomainError + sentinels (codes off error_prefix)
        ├── application/                        # use cases + ports (the only business logic)
        │   ├── port/                           # repository + transaction-boundary abstract base classes
        │   └── usecase/                        # use-case classes (throw DomainError)
        └── infrastructure/
            ├── grpc/                           # gRPC: <Foo>Service : <Foo>::Service   (gRPC profile)
            │   └── interceptor/                #   grpc::ServerInterceptor (auth, logging, ctx-id)
            ├── http/                           # HTTP: Drogon controllers (HTTP profile)
            ├── repositories/                   # mongocxx docs / SQL repo adapters (hand-written)
            └── config/                         # wiring (composition root)

protobuf/proto/<module>/...                     # SAME proto contracts as the other profiles — language-neutral
docs/openapi.yaml                               # emitted by the pipeline from the protos (do not hand-write)
```

Entrypoint per service: `main.cpp` — for gRPC, builds the `grpc::ServerBuilder` and starts the `grpc::Server`; for HTTP, registers Drogon controllers and calls `drogon::app().run()`. One service = one runnable binary.

## A.3 Hexagonal package layout (the C++ equivalents)

- **Layering & dependency rule:** identical to the Canon. The four roots are `domain`, `application`, `infrastructure/{grpc|http}`, `infrastructure/repositories`. Dependencies point **inward only**.
- **`domain`** — **framework-free** structs/type-aliases that wrap the proto-generated message classes, plus typed errors and constants. **The domain layer MUST NOT include mongocxx, libpqxx, the grpc++ server, or Drogon.** No `mongocxx::collection`, no `pqxx::connection`, no `grpc::*`. Domain is plain C++. This is the single hard rule that keeps the core portable.
- **`application`** — use cases (`application/usecase`) and ports (`application/port`). Use cases depend only on `domain` + the port interfaces. **Ports = abstract base classes** (`IFooRepository`, `ITransactionManager` with pure-virtual methods). Adapters in `infrastructure/repositories` implement them.
- **`infrastructure/{grpc|http}`** — the driving/transport adapters: the `<Foo>::Service` impls (gRPC profile) OR the Drogon controllers (HTTP profile). Translate transport ⇄ domain, delegate to the use case, map errors.
- **`infrastructure/repositories`** — the persistence adapters: mongocxx document mappers / SQL row mappers **plus** the repository adapter classes implementing the domain ports by mapping domain ⇄ persistence model.
- **Dependency injection = constructor injection**, assembled in `infrastructure/config` (the composition root). No statics/globals reaching into domain/application.
- **Domain types wrap the proto** (structs/aliases mirroring the proto messages) — never make them bson documents or SQL rows. The proto stays the single source of truth; transport objects and persistence models are **separate** types that map to/from the domain types.

## A.4 Error model (C++ mechanism for the Canon's taxonomy)

- A `DomainError` class carrying `Code` (string) and `Message`, with the same `Failure_Noun_Descriptor` message format and the orchestrator-assigned prefix (e.g. `IDN201`). Sentinel factories live in `domain/error` (e.g. `DomainError::NotFound(...)`, `DomainError::AlreadyExists(...)`). The application throws/returns `DomainError`; it never throws a bare `std::runtime_error`/`std::invalid_argument` for domain conditions.
- **gRPC mapping:** a shared `MapError(const DomainError&) -> grpc::Status` (e.g. `grpc::StatusCode::NOT_FOUND`, `ALREADY_EXISTS`, `FAILED_PRECONDITION`, `INVALID_ARGUMENT`, `INTERNAL`). `1xx` → `INVALID_ARGUMENT`, `2xx` domain codes → the domain-specific status, `5xx` → `INTERNAL`. Every new code handled explicitly. The `<Foo>::Service` method (or an interceptor) translates the thrown/returned `DomainError`.
- **HTTP mapping (Drogon):** a shared mapper translating each `Code` to an HTTP status (`1xx` → 400, not-found → 404, forbidden → 403, conflict → 409, `5xx` → 500) returning a JSON body `{ code, message }`.
- Direct `INVALID_ARGUMENT` / HTTP 400 is allowed **only** for request-field validation before delegating to the use case.

## A.4-HTTP HTTP transport rule (CRITICAL — no gRPC server in HTTP services)

For an **HTTP service** the **Drogon app is the ONLY entrypoint**. The authoritative `.proto` is still written WITH `google.api.http` annotations on every RPC (it drives `docs/openapi.yaml`), **but at RUNTIME there is NO gRPC server**:

- **There MUST be no `grpc::ServerBuilder`, no `grpc::Server`, no `*.grpc.pb.*` service base wired, and no `infrastructure/grpc/` directory.** The protoc custom command emits ONLY message classes (`--cpp_out`, **no** `--grpc_out`).
- Handlers are Drogon controller methods, **1:1 with the RPCs**, mounted on the verb + path declared by each RPC's `google.api.http` annotation.
- Request/response messages are the proto-generated message classes (or DTOs mapping to them), serialized as JSON — not marshalled through a gRPC server.

> **Note:** in this profile the HTTP/Drogon cell is *filled* but the certified cell is **gRPC**. Follow the Drogon patterns when the boundary spec selects HTTP; the gRPC path is the reference path.

## A.5 Persistence — MongoDB (default; mongocxx — ARTISANAL, no ORM)

- **Driver:** `mongocxx` / `bsoncxx`. For each owned resource define a repository adapter `MongoFooRepository : IFooRepository` in `infrastructure/repositories` (NOT in domain) implementing the port **BY HAND**, mapping domain struct ⇄ bson document over a `mongocxx::collection`. The domain types are structs/type-aliases of the generated `*.pb.h` message classes. There is **NO ORM**.
- **IDs:** the `system_counters` pattern — a dedicated collection, an atomic `find_one_and_update` `$inc` (return after, upsert), seeded at a reserved start value, stored as BSON `int64` and mapped to the proto `uint64 identifier`. (For SQL, IDs come from an autoincrement/identity PK instead — §A.13.)
- **Soft-delete:** a nullable `delete_time` field; deletes set it, reads filter it out (`delete_time == null`).
- **Config:** `MONGO_URI` / `MONGO_DATABASE` (read via `std::getenv`). Use a `mongocxx::instance` + `mongocxx::client`/`mongocxx::pool`. **No SQL libraries when store is mongodb.**

## A.6 Transaction boundary (port + adapter)

- An `ITransactionManager` port with a `WithTransaction(std::function<void()> body)` method.
- Mongo adapter implements it over a driver session (`mongocxx::client_session` + `with_transaction`); SQL adapter implements it over a `pqxx::work` (or the MySQL connector transaction). A service needing no boundary omits the dependency.

## A.7 Logging

- One logger mechanism: a single injected logging facility (the Drogon `LOG_*` macros for HTTP, or a thin `spdlog`/`std::clog`-backed logger injected via the composition root for gRPC). Structured format `"<context>: key=value"`. **Forbidden:** raw `std::cout`/`printf` scattered in layers, or any ad-hoc logging. Everything goes through the injected logger.

## A.8 Testing (C++ mechanism for the Canon's philosophy)

- **GoogleTest** (`find_package(GTest)`) when available, else a **minimal assert-based test main**. Application unit tests run via `ctest`. The application layer is tested against its ports using test doubles (hand-rolled fakes implementing the port abstract base classes); mongocxx/libpqxx are **not** mocked in application tests — the application layer only sees ports.
- Coverage floor (Canon §8): every use case has a success test + ≥1 error scenario; every error sentinel asserted by ≥1 test. Test names `<Operation>_<Scenario>`.
- **`ctest` is the hard build gate** — it must pass with zero failures. The whole workspace must also compile via **`cmake --build`** (which includes the protoc/`grpc_cpp_plugin` codegen producing `*.pb.cc`/`*.grpc.pb.cc`).

## A.9 C++-specific verifier gates

In addition to the Canon-level gates (`buf lint`, AIP, OpenAPI emission):

- **`cmake --build` succeeds** with zero errors (the protoc custom command produces the message + service stubs), and **`ctest` passes** with zero failures (the primary gate). For gRPC services the build includes the `grpc_cpp_plugin` codegen producing `*.grpc.pb.cc` that compiles.
- **Layer-import discipline** — encode the dependency rule via `#include`/target boundaries: `domain` includes only the standard library + the proto message headers (NO mongocxx/libpqxx/grpc++ server/Drogon); `application` includes only `domain` + its ports; `infrastructure/{grpc|http}` never includes `repositories` directly. A violation is caught in the self-audit.
- **No hand-edited generated protobuf/grpc code** (`*.pb.*`/`*.grpc.pb.*` from protoc are build output in the build dir, never committed/shipped).
- **HTTP services:** zero `grpc::ServerBuilder` / `grpc::Server` / `infrastructure/grpc/` (A.4-HTTP).

## A.10 What stays a hole / Cleanliness

- **PostgreSQL / MySQL/MariaDB — FILLED (see §A.13).** When `store: postgres` or `store: mysql` the generator emits the artisanal parametrized-SQL layer described in §A.13.
- **NATS and any non-gRPC/non-HTTP transport** remain holes. If a task or boundary spec requires them, **stop and write a blocker note** — do not improvise.
- **Cleanliness (MUST):** never ship `build/`, `CMakeFiles/`, `*.o`, `*.a`, `*.so`, or `CMakeCache.txt` in the deliverable (the assembler drops them defensively, but do not write them either). No platform-proto leak beyond the import closure actually used. Idiomatic C++ naming (`PascalCase` types, `snake_case` functions/locals, per idiomatic C++). Add a `.gitignore` ignoring `build/` if warranted.

## A.11 Build-cost note (containerised gate)

C++ compile + link of grpc++/mongocxx is heavy → the C++ profile runs on the **heavy resource tier** (4 GiB / 2 CPU / 90 min), like rust, java and csharp. BUT because grpc++/protobuf/mongocxx are **PREINSTALLED via apt** and resolved with `pkg_check_modules` (pkg-config, NOT FetchContent), the runtime build only compiles the service's own code + links against the preinstalled `.so` libraries — **keep it that way**. To stay within the container budget: resolve every dependency with the certified `pkg_check_modules` (pkg-config) — or `find_package(CONFIG)` where one exists — against the preinstalled libs (§A.1.1, golden template §A.1.1.1), NEVER FetchContent/ExternalProject grpc/protobuf/mongocxx (re-downloading and recompiling them blows the timeout), and keep the dependency set minimal.

## A.12 Auth / Validation

JWT is the only request-auth scheme generated in v1 (besides `none`). When the generation prompt's "Auth / Validation: JWT" section is present, implement bearer JWT validation with **jwt-cpp** (header-only, `#include <jwt-cpp/jwt.h>`):

- Read the token from `Authorization: Bearer <token>`; read the secret/public key, issuer, audience and required claims from `ENV` (`JWT_SECRET`, `JWT_ISSUER`, `JWT_AUDIENCE`), NEVER hardcoded.
- **HTTP:** a **Drogon filter** (`drogon::HttpFilter`) validating the token on the protected endpoints; expose the authenticated identity (e.g. `sub`) to handlers via the request attributes.
- **gRPC:** a **`grpc::ServerInterceptor`** (implementing `grpc::experimental::Interceptor` via a `grpc::experimental::ServerInterceptorFactoryInterface`) that reads the `authorization` metadata, verifies with jwt-cpp (`jwt::verify`), and places the identity in the call context.
- Reject `alg=none` and any algorithm outside the configured family (HS* symmetric; RS*/ES*/EdDSA asymmetric public key) — defend against algorithm confusion. On failure map to HTTP 401 / gRPC `UNAUTHENTICATED`, without leaking the reason or the token.
- The validation must compile under `cmake --build` and be covered by a `ctest` test.

A detected scheme other than JWT (OAuth2 authorization-code / session cookie / API key / Basic) is a hole in v1: do NOT improvise it. Generate without an auth layer and add a TODO — or re-run with `target_auth_scheme = AUTH_SCHEME_JWT`.

## A.13 Infrastructure — SQL adapters via artisanal parametrized SQL (`store: postgres` | `store: mysql`)

When the boundary spec carries `store: postgres` or `store: mysql` (and the generation prompt's "Persistence: … (artisanal parametrized SQL)" section is present), generate a **hand-written parametrized-SQL** persistence layer **instead of** §A.5–A.6's mongocxx adapters. This is the C++ homologue of the Go-GORM, Python-SQLAlchemy, Node-Prisma, Rust-SeaORM, Java-JPA, Ruby-ActiveRecord and C#-EF Core SQL cells — **but artisanal**: there is NO ORM; the repository implements the port BY HAND with parametrized SQL. The SAME repository shape serves PostgreSQL AND MySQL/MariaDB — only the driver (`libpqxx` vs `mysql-connector-c++`), the parameter syntax and the connection string differ. Domain types remain framework-free structs (Canon §5.1, A.3), never SQL rows — the row-mapping is **separate**.

- **Driver.** Use exactly one: `libpqxx` (`pqxx::connection`, parametrized `exec_params`) for `store: postgres`, `mysql-connector-c++` (prepared statements) for `store: mysql`. Do NOT add mongocxx on a SQL store; do NOT add both drivers.
- **Repositories.** A `SqlFooRepository : IFooRepository` adapter per owned resource mapping domain ⇄ row with **parametrized** SQL (never string-concatenated). Only the implementation changes; application/handlers are unchanged.
- **Identifier generation (Canon §5.3).** Autoincrement/identity primary key by the database (`SERIAL`/`BIGSERIAL` for Postgres, `BIGINT AUTO_INCREMENT` for MySQL) — never an emulated `system_counters` table. Map the proto `uint64 identifier` to/from the row's `BIGINT` without losing precision.
- **Schema.** A hand-written `schema.sql` file (NOT autogenerated — there is no ORM migration) deriving the schema from the boundary spec's `owned_resources`. FK columns/indexes come from `cross_service_fks` (FK columns/indexes only — never a hard cross-service FK constraint, per the data-ownership boundary).
- **Soft-delete.** A nullable `delete_time` column; deletes set it (update), reads filter `delete_time IS NULL`, matching the Mongo path's semantics.
- **Transaction boundary.** A `pqxx::work`-backed (or MySQL-connector-transaction-backed) `ITransactionManager` adapter; a service needing no boundary omits it.
- **Config.** Read from `ENV` (`std::getenv`): the connection string (the full driver form — `Host=…`/`postgresql://…` for libpqxx, the MySQL DSN otherwise). **Never** hardcode a password; emit zero `MONGO_*` keys. **No Mongo libraries when SQL and vice-versa.** The assembler ships a per-service `.env.example` with these keys and the hand-written `schema.sql`.
- **Testing.** Application tests still use test doubles of the port interfaces (no DB). Repository round-trips may be exercised against a container-backed DB if available; `cmake --build` + `ctest` is the gate (A.8/A.9).
