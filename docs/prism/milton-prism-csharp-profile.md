# Milton Prism — C# (.NET / grpc-dotnet / ASP.NET Core) Language Profile

Peer document to `milton-prism-go-profile.md`, `milton-prism-python-profile.md`, `milton-prism-node-profile.md`, `milton-prism-rust-profile.md`, `milton-prism-java-profile.md` and `milton-prism-ruby-profile.md`. This is the canonical **C# + .NET + grpc-dotnet / ASP.NET Core** mechanism reference for all C# services generated by Milton Prism. It composes with the **Architecture Canon** (`milton-prism-architecture-canon.md`) exactly as the other Profiles do: the Canon's principles, the proto/AIP contract, the error taxonomy, the dependency rule, and the testing philosophy all still apply unchanged. This document fills only the C# *mechanisms*.

> **Scope note:** this Profile defines the *output* language for generated microservices. It is independent of the analysis engine's input languages. Supported stores are **MongoDB** (MongoDB.Driver, §A.5–A.6) **and SQL — PostgreSQL / MySQL/MariaDB — via Entity Framework Core (§A.13)**; transports are **gRPC** (grpc-dotnet) **and HTTP** (ASP.NET Core Minimal API). Branch on the boundary spec's `store:` field: `store: mongodb` → the MongoDB.Driver collections/repos; `store: postgres` OR `store: mysql` → §A.13's EF Core DbContext/repos. **NATS** remains a hole — if requested, stop and write a blocker note rather than improvising.

> **No reference implementation in the repo.** Unlike the Python profile (which ships a `python/` skeleton), the C# deliverable is generated **entirely from this document plus the proto contract**. There is no pre-existing `csharp/` tree to read. Follow the patterns in this document literally; they ARE the reference.

---

## A.1 Stack (fixed)

- **Language / runtime:** **C# on .NET 9** (the agent image ships the Microsoft .NET SDK 9, glibc). Nullable reference types enabled. The whole workspace must pass `dotnet build` with zero errors and the suite must pass `dotnet test`.
- **Dependency manager / build:** **NuGet + MSBuild** via the **.NET SDK**. One **`.csproj`** per service under `csharp/services/<service>/` declares the package references; a solution file (`.sln`) at the source root may tie them together but is optional. **Build gate: `dotnet build` (restores from the warmed NuGet cache) + `dotnet test`.**
- **Dependency hygiene:** each `.csproj` declares ONLY the `PackageReference`s the code actually uses — no unused/leftover package; keep the set minimal.
- **Warmed NuGet cache.** The agent image ships a pre-populated, world-readable NuGet package cache (`NUGET_PACKAGES`, a warmed `dotnet restore` of the A.1.1 set plus the Roslyn analyzer graph). The generated `.csproj` **MUST use only packages from the warmed set** at compatible versions — adding an un-warmed package will fail/stall the offline-leaning `dotnet restore`. Keep the package set minimal. **Never** ship `bin/`, `obj/`, `*.dll`, `*.pdb`, or `*.nupkg` in the deliverable (see A.10 Cleanliness).
- **gRPC:** **grpc-dotnet** — `Grpc.AspNetCore` hosting the service via `builder.Services.AddGrpc()` + `app.MapGrpcService<FooService>()` on Kestrel (HTTP/2). The service class **subclasses the generated `Foo.FooBase`**. The message + service stubs are generated at build time by **`Grpc.Tools`** (`protoc` + `grpc_csharp_plugin`, both glibc) from `<Protobuf>` items in the `.csproj` — `*.cs` message classes + `*Grpc.cs` service base.
- **HTTP:** **ASP.NET Core Minimal API** — `WebApplication` mapping endpoints, the only entrypoint. (See A.4-HTTP and the HTTP generation prompt.)
- **Proto codegen:** the `.csproj` `<Protobuf Include="...protobuf/proto/..." GrpcServices="Server" />` items run `Grpc.Tools` against the canonical `protobuf/proto` include root; generated C# lands in `obj/` (build intermediate) and is **NEVER edited by hand** and **NEVER shipped**.
- **Mongo driver:** **MongoDB.Driver** (`MongoDB.Driver` NuGet) — `IMongoCollection<TDocument>` document classes, repositories on top.
- **SQL:** **Entity Framework Core** (`Microsoft.EntityFrameworkCore` + `Npgsql.EntityFrameworkCore.PostgreSQL` / `Pomelo.EntityFrameworkCore.MySql`) — a `DbContext` + POCO entities, repositories (§A.13).
- **Config:** environment variables (`MONGO_URI`, `MONGO_DATABASE`, `ConnectionStrings__Default`, `GRPC_HOST`, `GRPC_PORT`, `JWT_SECRET`) loaded via `IConfiguration` / `Environment.GetEnvironmentVariable`. Never hardcode secrets.
- **IDs:** Mongo → BSON `Int64` (C# `long`) for the proto `uint64 identifier` field, never narrowed. SQL → autoincrement primary key (the default EF Core identity `long` PK).

### A.1.1 Pinned package set (must match the warmed NuGet cache)

The generated `.csproj` MUST use exactly these NuGet packages (kept in lockstep with the agent-image author):

| Purpose | Package |
|---------|---------|
| gRPC server (ASP.NET Core) | `Grpc.AspNetCore` |
| gRPC codegen (protoc + plugin) | `Grpc.Tools` |
| Protobuf runtime | `Google.Protobuf` |
| ASP.NET Core (framework ref) | `Microsoft.AspNetCore.App` (shared framework) |
| Mongo store | `MongoDB.Driver` |
| SQL store (ORM) | `Microsoft.EntityFrameworkCore` |
| PostgreSQL provider | `Npgsql.EntityFrameworkCore.PostgreSQL` |
| MySQL/MariaDB provider | `Pomelo.EntityFrameworkCore.MySql` |
| Auth (JWT) | `Microsoft.AspNetCore.Authentication.JwtBearer` (or `System.IdentityModel.Tokens.Jwt`) |
| Tests | `Microsoft.NET.Test.Sdk` + `xunit` + `xunit.runner.visualstudio` |

> **Do not introduce any package not in this table.** If a needed package is missing from the warmed cache, stop and report an environment blocker rather than reaching to the network. Use the Mongo package for `store: mongodb`, the EF Core + provider packages for `store: postgres|mysql` — never both store stacks at once.

## A.2 Repository layout (mirrors the Go/Python/Node/Rust/Java/Ruby monorepo conceptually)

The agent writes everything under the source root **`csharp/`** (the assembler renames `csharp/` → `core/` in the final deliverable, exactly as Node renames `node/` → `core/`, Java renames `java/` → `core/` and Ruby renames `ruby/` → `core/`). Protos and docs are siblings of the source root.

```
csharp/                                         # source root (→ core/ in deliverable)
└── services/
    └── <service>/
        ├── <Service>.csproj                    # package refs (A.1.1) + <Protobuf> items
        ├── Program.cs                          # entrypoint: build & start the Kestrel gRPC host
        ├── Domain/                             # framework-free types + typed errors
        │   ├── Model/                          # plain C# records/classes wrapping the proto messages
        │   └── Error/                          # DomainError + sentinels (codes off error_prefix)
        ├── Application/                        # use cases + ports (the only business logic)
        │   ├── Port/                           # repository + transaction-boundary interfaces
        │   └── UseCase/                        # use-case classes (throw DomainError)
        └── Infrastructure/
            ├── Grpc/                           # gRPC: <Foo>Service : Foo.FooBase   (gRPC profile)
            │   └── Interceptor/                #   grpc-dotnet Interceptor (auth, logging, ctx-id)
            ├── Http/                           # HTTP: Minimal API endpoint mappers (HTTP profile)
            ├── Repositories/                   # MongoDB.Driver docs / EF Core DbContext + entities + repo adapters
            └── Config/                         # wiring (composition root / DI registration)

protobuf/proto/<module>/...                     # SAME proto contracts as the other profiles — language-neutral
docs/openapi.yaml                               # emitted by the pipeline from the protos (do not hand-write)
```

Entrypoint per service: `Program.cs` — for gRPC, builds the `WebApplication` and `MapGrpcService<>` on Kestrel; for HTTP, maps Minimal API endpoints. One service = one runnable app.

## A.3 Hexagonal package layout (the C# equivalents)

- **Layering & dependency rule:** identical to the Canon. The four roots are `Domain`, `Application`, `Infrastructure/{Grpc|Http}`, `Infrastructure/Repositories`. Dependencies point **inward only**.
- **`Domain`** — **framework-free** records/classes that wrap the proto-generated message classes, plus typed errors and constants. **The domain layer MUST NOT reference MongoDB.Driver, EF Core, the gRPC server, or ASP.NET Core.** No `IMongoCollection`, no `DbContext`, no `Grpc.Core.*`. Domain is plain C#. This is the single hard rule that keeps the core portable.
- **`Application`** — use cases (`Application/UseCase`) and ports (`Application/Port`). Use cases depend only on `Domain` + the port interfaces. **Ports = C# interfaces** (`IFooRepository`, `ITransactionManager`). Adapters in `Infrastructure/Repositories` implement them.
- **`Infrastructure/{Grpc|Http}`** — the driving/transport adapters: the `Foo.FooBase` service impls (gRPC profile) OR the Minimal API endpoint mappers (HTTP profile). Translate transport ⇄ domain, delegate to the use case, map errors.
- **`Infrastructure/Repositories`** — the persistence adapters: MongoDB document classes / EF Core entity classes + the `DbContext` **plus** the repository adapter classes implementing the domain ports by mapping domain ⇄ persistence model.
- **Dependency injection = constructor injection**, assembled in `Infrastructure/Config` (the composition root, registered on `IServiceCollection`). No statics reaching into domain/application.
- **Domain types wrap the proto** (records mirroring the proto messages) — never make them MongoDB documents or EF Core entities. The proto stays the single source of truth; transport objects and persistence models are **separate** classes that map to/from the domain types.

## A.4 Error model (C# mechanism for the Canon's taxonomy)

- A `DomainError` class (`: Exception`) carrying `Code` (string) and `Message`, with the same `Failure_Noun_Descriptor` message format and the orchestrator-assigned prefix (e.g. `IDN201`). Sentinel factories live in `Domain/Error` (e.g. `DomainError.NotFound(...)`, `DomainError.AlreadyExists(...)`). The application throws `DomainError`; it never throws a bare `InvalidOperationException`/`ArgumentException` for domain conditions.
- **gRPC mapping:** a shared `MapError(DomainError) -> RpcException` (e.g. `StatusCode.NotFound`, `AlreadyExists`, `FailedPrecondition`, `InvalidArgument`, `Internal`). `1xx` → `InvalidArgument`, `2xx` domain codes → the domain-specific status, `5xx` → `Internal`. Every new code handled explicitly. A grpc-dotnet `Interceptor` (or per-method `try/catch`) translates the thrown `DomainError`.
- **HTTP mapping:** an exception-handling middleware (or `Results.Problem`) translating each `Code` to an HTTP status (`1xx` → 400, not-found → 404, forbidden → 403, conflict → 409, `5xx` → 500) returning a JSON body `{ code, message }`.
- Direct `InvalidArgument` / HTTP 400 is allowed **only** for request-field validation before delegating to the use case.

## A.4-HTTP HTTP transport rule (CRITICAL — no gRPC server in HTTP services)

For an **HTTP service** the **ASP.NET Core app is the ONLY entrypoint**. The authoritative `.proto` is still written WITH `google.api.http` annotations on every RPC (it drives `docs/openapi.yaml`), **but at RUNTIME there is NO gRPC server**:

- **There MUST be no `AddGrpc()`, no `MapGrpcService<>`, no `*Grpc.cs` service base wired, and no `Infrastructure/Grpc/` package.** The `.csproj` `<Protobuf>` items use `GrpcServices="None"` (message classes only).
- Handlers are Minimal API endpoints (`app.MapGet`/`MapPost`/...), **1:1 with the RPCs**, mounted on the verb + path declared by each RPC's `google.api.http` annotation.
- Request/response messages are the proto-generated message classes (or DTOs mapping to them), serialized as JSON — not marshalled through a gRPC server.

## A.5 Persistence — MongoDB (default; MongoDB.Driver)

- **Driver:** `MongoDB.Driver`. For each owned resource define a document class in `Infrastructure/Repositories` (NOT in domain) and a repository adapter `MongoFooRepository : IFooRepository`, mapping domain record ⇄ document via an `IMongoCollection<FooDocument>`.
- **IDs:** the `system_counters` pattern — a dedicated collection, an atomic `FindOneAndUpdate` `$inc` (`ReturnDocument.After`, `IsUpsert = true`), seeded at a reserved start value, stored as BSON `Int64` and mapped to the proto `uint64 identifier`. (For SQL, IDs come from an EF Core identity PK instead — §A.13.)
- **Soft-delete:** a nullable `DeleteTime` field; deletes set it, reads filter it out (`DeleteTime == null`).
- **Config:** `MONGO_URI` / `MONGO_DATABASE` (read via `IConfiguration` / `ENV`, the `IMongoClient`/`IMongoDatabase` registered in DI). **No EF Core packages when store is mongodb.**

## A.6 Transaction boundary (port + adapter)

- An `ITransactionManager` port with a `WithTransactionAsync(Func<Task> body)` method.
- Mongo adapter implements it over a driver session (`IClientSessionHandle` + `WithTransactionAsync`); SQL adapter implements it over `DbContext.Database.BeginTransactionAsync`. A service needing no boundary omits the dependency.

## A.7 Logging

- One logger mechanism: **`Microsoft.Extensions.Logging`** (`ILogger<T>` injected via DI). Structured format `"<context>: key=value"`. **Forbidden:** `Console.WriteLine` scattered in layers, or any ad-hoc logging. Everything goes through the injected `ILogger`.

## A.8 Testing (C# mechanism for the Canon's philosophy)

- **xUnit** (`xunit` + `Microsoft.NET.Test.Sdk`). Application unit tests run via `dotnet test`. The application layer is tested against its ports using test doubles (hand-rolled fakes or `Moq` if warmed) of the port interfaces; MongoDB.Driver/EF Core are **not** mocked in application tests — the application layer only sees ports.
- Coverage floor (Canon §8): every use case has a success test + ≥1 error scenario; every error sentinel asserted by ≥1 test. Test method names `<Operation>_<Scenario>`.
- **`dotnet test` is the hard build gate** — it must pass with zero failures. The whole workspace must also compile via **`dotnet build`** (which includes the `Grpc.Tools` codegen producing `*.cs`/`*Grpc.cs`).

## A.9 C#-specific verifier gates

In addition to the Canon-level gates (`buf lint`, AIP, OpenAPI emission):

- **`dotnet build` succeeds** with zero errors (the codegen `<Protobuf>` items produce the message + service stubs), and **`dotnet test` passes** with zero failures (the primary gate). For gRPC services the build includes the `Grpc.Tools` codegen producing `*Grpc.cs` that compiles.
- **Layer-import discipline** — encode the dependency rule via `using`/project boundaries: `Domain` references only the BCL + the proto message classes (NO MongoDB.Driver/EF Core/gRPC server); `Application` references only `Domain` + its ports; `Infrastructure/{Grpc|Http}` never references `Repositories` directly. A violation is caught in the self-audit.
- **No hand-edited generated protobuf/grpc code** (`*.cs`/`*Grpc.cs` from `Grpc.Tools` are build output under `obj/`, never committed/shipped).
- **HTTP services:** zero `AddGrpc()` / `MapGrpcService` / `Infrastructure/Grpc/` (A.4-HTTP).

## A.10 What stays a hole / Cleanliness

- **PostgreSQL / MySQL/MariaDB — FILLED (see §A.13).** When `store: postgres` or `store: mysql` the generator emits the EF Core layer described in §A.13.
- **NATS and any non-gRPC/non-HTTP transport** remain holes. If a task or boundary spec requires them, **stop and write a blocker note** — do not improvise.
- **Cleanliness (MUST):** never ship `bin/`, `obj/`, `*.dll`, `*.pdb`, or `*.nupkg` in the deliverable (the assembler drops them defensively, but do not write them either). No platform-proto leak beyond the import closure actually used. Idiomatic C# naming (PascalCase types/methods, camelCase locals). Add a `.gitignore` ignoring `bin/`, `obj/` if warranted.

## A.11 Build-cost note (containerised gate)

C# compiles via Roslyn and `dotnet build` runs the NuGet restore graph + Roslyn analyzers + the `Grpc.Tools` protoc codegen — this peaks **above 1 GiB**, so the C# profile runs on the **heavy resource tier** (4 GiB / 2 CPU / 90 min), like rust and java. To stay within the container budget: keep the package set minimal (only A.1.1's table), lean on the pre-warmed NuGet cache to absorb the restore (`dotnet build` should restore offline), and avoid heavy optional packages beyond what the service needs.

## A.12 Auth / Validation

JWT is the only request-auth scheme generated in v1 (besides `none`). When the generation prompt's "Auth / Validation: JWT" section is present, implement bearer JWT validation with **`Microsoft.AspNetCore.Authentication.JwtBearer`** (HTTP) or **`System.IdentityModel.Tokens.Jwt`** (gRPC interceptor):

- Read the token from `Authorization: Bearer <token>`; read the secret/public key, issuer, audience and required claims from `IConfiguration`/`ENV` (`JWT_SECRET`, `JWT_ISSUER`, `JWT_AUDIENCE`), NEVER hardcoded.
- **HTTP:** wire `AddAuthentication().AddJwtBearer(...)` + `UseAuthentication()`/`UseAuthorization()`; expose the authenticated identity (e.g. `sub`) to handlers via `HttpContext.User`.
- **gRPC:** wire as a **grpc-dotnet `Interceptor`** that validates the token from the call metadata (`ServerCallContext.RequestHeaders`) and places the identity in the call context.
- Reject `alg=none` and any algorithm outside the configured family (HS* symmetric; RS*/ES*/EdDSA asymmetric public key). On failure throw a typed error mapped to HTTP 401 / gRPC `Unauthenticated`, without leaking the reason or the token.
- The validation must compile under `dotnet build` and be covered by an xUnit test.

A detected scheme other than JWT (OAuth2 authorization-code / session cookie / API key / Basic) is a hole in v1: do NOT improvise it. Generate without an auth layer and add a TODO — or re-run with `target_auth_scheme = AUTH_SCHEME_JWT`.

## A.13 Infrastructure — SQL adapters via Entity Framework Core (`store: postgres` | `store: mysql`)

When the boundary spec carries `store: postgres` or `store: mysql` (and the generation prompt's "Persistence: … (Entity Framework Core)" section is present), generate an **EF Core** persistence layer **instead of** §A.5–A.6's MongoDB.Driver adapters. This is the C# homologue of the Go-GORM, Python-SQLAlchemy, Node-Prisma, Rust-SeaORM, Java-JPA and Ruby-ActiveRecord SQL cells: **the SAME `DbContext` + entity set serves PostgreSQL AND MySQL/MariaDB** — only the provider package + the `UseXxx` call + the connection string differ (`Host=…` Npgsql vs `Server=…` Pomelo). Domain types remain framework-free records (Canon §5.1, A.3), never EF Core entities — the entity classes are **separate**.

- **Provider package.** Add `Microsoft.EntityFrameworkCore` PLUS exactly one provider: `Npgsql.EntityFrameworkCore.PostgreSQL` (`UseNpgsql`) for `store: postgres`, `Pomelo.EntityFrameworkCore.MySql` (`UseMySql` — it speaks both MySQL and MariaDB) for `store: mysql`. Do NOT add `MongoDB.Driver` on a SQL store; do NOT add both providers.
- **Entities live in infrastructure (NOT domain).** For each owned resource define a POCO entity in `Infrastructure/Repositories` (snake_case table/columns via Fluent API), the PK the default `long` identity. Domain records are NEVER EF Core entities.
- **Repositories.** An `EfFooRepository : IFooRepository` adapter per owned resource mapping domain ⇄ entity. Only the implementation changes; application/handlers are unchanged.
- **Identifier generation (Canon §5.3).** Identity primary key by the database (`ValueGeneratedOnAdd()`) — never an emulated `system_counters` table. Map the proto `uint64 identifier` to/from the entity's `long` without losing precision.
- **Schema.** EF Core migrations (`dotnet ef migrations`, or `EnsureCreated` in tests) derive the schema from the entities (themselves derived from the boundary spec's `owned_resources`). FK columns/indexes come from `cross_service_fks` (FK columns/indexes only — never a hard cross-service FK constraint, per the data-ownership boundary).
- **Soft-delete.** A nullable `DeleteTime` column; deletes set it (update), reads filter `DeleteTime == null` (a global query filter is acceptable), matching the Mongo path's semantics.
- **Transaction boundary.** A `DbContext.Database.BeginTransactionAsync`-backed `ITransactionManager` adapter; a service needing no boundary omits it.
- **Config.** Read from `IConfiguration`/`ENV`: the connection string (`ConnectionStrings__Default`, the full ADO.NET form — `Host=…` or `Server=…`). **Never** hardcode a password; emit zero `MONGO_*` keys. **No Mongo package when SQL and vice-versa.** The assembler ships a per-service `.env.example` with these keys.
- **Testing.** Application tests still mock the port interfaces (no DB). Repository round-trips may be exercised with the EF Core Sqlite/InMemory provider (if warmed) or a container-backed test; `dotnet build` + `dotnet test` is the gate (A.8/A.9).
