# Milton Prism — Java (Spring Boot / grpc-java) Language Profile

Peer document to `milton-prism-go-profile.md`, `milton-prism-python-profile.md`, `milton-prism-node-profile.md`, and `milton-prism-rust-profile.md`. This is the canonical **Java + Maven + Spring Boot + gRPC/HTTP** mechanism reference for all Java services generated by Milton Prism. It composes with the **Architecture Canon** (`milton-prism-architecture-canon.md`) exactly as the Go/Python/Node/Rust Profiles do: the Canon's principles, the proto/AIP contract, the error taxonomy, the dependency rule, and the testing philosophy all still apply unchanged. This document fills only the Java *mechanisms*.

> **Scope note:** this Profile defines the *output* language for generated microservices. It is independent of the analysis engine's input languages. Supported stores are **MongoDB** (Spring Data MongoDB, §A.5–A.6) **and SQL — PostgreSQL / MySQL/MariaDB — via Spring Data JPA / Hibernate (§A.13)**; transports are **gRPC** (grpc-java) **and HTTP** (Spring Boot `@RestController`). Branch on the boundary spec's `store:` field: `store: mongodb` → the Spring Data MongoDB adapters; `store: postgres` OR `store: mysql` → §A.13's Spring Data JPA adapters. **NATS** remains a hole — if requested, stop and write a blocker note rather than improvising.

> **No reference implementation in the repo.** Unlike the Python profile (which ships a `python/` skeleton), the Java deliverable is generated **entirely from this document plus the proto contract**. There is no pre-existing `java/` tree to read. Follow the patterns in this document literally; they ARE the reference.

---

## A.1 Stack (fixed)

- **Language / runtime:** **Java 21 LTS** (JDK 21). `<maven.compiler.release>21</maven.compiler.release>`; records and sealed types are encouraged for framework-free domain POJOs. The build must compile with zero errors (`mvn -B package`).
- **Build system:** **Maven** (Apache Maven 3.9+). A single **parent `pom.xml`** at the source root (`java/pom.xml`) declares `<packaging>pom</packaging>`, the dependency/version management (`<dependencyManagement>` pinned to the Spring Boot BOM), the plugin versions, and `<modules>` — **one module per service** under `java/services/<service>/`. Each service module is its own `pom.xml` inheriting the parent. **Build gate: `mvn -B package`; tests: `mvn -B test`.**
- **Dependency hygiene:** each service `pom.xml` declares ONLY the dependencies that module's code actually uses — no unused/leftover `<dependency>`; keep the set minimal.
- **Warmed local repository.** The agent image ships a pre-populated Maven local repository at **`/usr/local/m2`** (configured via `-Dmaven.repo.local=/usr/local/m2` or `settings.xml`). The generated POMs **MUST use only dependencies from the warmed set** (listed in A.1.1) at the pinned versions — adding an un-warmed dependency or a different version will fail/stall the offline-leaning build. Keep the dependency set minimal. **Never** vendor a `.m2` into the deliverable (see A.10 Cleanliness).
- **gRPC:** **grpc-java** — the async/blocking gRPC server and inter-service stubs. gRPC is the primary transport per the Canon. Stubs are generated by `protobuf-maven-plugin` (+ `os-maven-plugin`) at build time; service impls extend the generated `*ImplBase`.
- **HTTP:** **Spring Boot** (`spring-boot-starter-web`) — `@SpringBootApplication` entrypoint with `@RestController` handlers. (See A.4-HTTP and the HTTP generation prompt.)
- **Proto codegen:** `protobuf-maven-plugin` invokes `protoc` against the canonical `protobuf/proto` include root; generated Java lands under `target/generated-sources/protobuf/` and is **NEVER edited by hand** and **NEVER committed**.
- **Mongo driver:** **Spring Data MongoDB** (`spring-boot-starter-data-mongodb`) — `MongoRepository`, `@Document`.
- **SQL:** **Spring Data JPA / Hibernate** (`spring-boot-starter-data-jpa`) — `@Entity`, `JpaRepository` (§A.13).
- **Config:** Spring `application.yml` + environment variables (`DATABASE_URL`, `SPRING_DATASOURCE_*`, `SPRING_DATA_MONGODB_URI`, `GRPC_PORT`/`SERVER_PORT`, `JWT_SECRET`). Never hardcode secrets.
- **IDs:** Mongo → BSON `Long` (`java.lang.Long`) for the proto `uint64 identifier` field, never narrowed. SQL → autoincrement primary key (`@GeneratedValue(strategy = GenerationType.IDENTITY)`).

## A.2 Repository layout (mirrors the Go/Python/Node/Rust monorepo conceptually)

The agent writes everything under the source root **`java/`** (the assembler renames `java/` → `core/` in the final deliverable, exactly as Node renames `node/` → `core/` and Rust renames `rust/` → `core/`). Protos and docs are siblings of the source root.

```
java/                                          # Maven reactor source root (→ core/ in deliverable)
├── pom.xml                                    # PARENT pom: packaging=pom, Spring Boot BOM import,
│                                              #   <dependencyManagement>, plugin mgmt, <modules>
└── services/
    └── <service>/
        ├── pom.xml                            # module pom (inherits parent); only this service's deps
        └── src/
            ├── main/
            │   ├── java/com/miltonprism/<service>/
            │   │   ├── Application.java                 # @SpringBootApplication entrypoint
            │   │   ├── domain/                          # framework-free POJOs/records + typed errors
            │   │   │   ├── model/                       # records derived from proto resources
            │   │   │   └── error/                       # DomainError + sentinels (codes off error_prefix)
            │   │   ├── application/                     # use cases + ports (the only business logic)
            │   │   │   ├── port/                        # repository + transaction-boundary interfaces
            │   │   │   └── usecase/                     # use-case services (Result/exception model)
            │   │   └── infrastructure/
            │   │       ├── grpc/                        # gRPC: service impls extending *ImplBase  (gRPC profile)
            │   │       │   └── interceptor/             #   ServerInterceptor (auth, logging, ctx-id)
            │   │       ├── http/                        # HTTP: @RestController + security filter  (HTTP profile)
            │   │       ├── repositories/                # JPA @Entity / Mongo @Document + Spring Data repos
            │   │       └── config/                      # @Configuration / wiring beans
            │   └── resources/
            │       └── application.yml                  # config (env-overridable)
            └── test/java/com/miltonprism/<service>/     # JUnit 5 unit tests (mvn -B test)

protobuf/proto/<module>/...                    # SAME proto contracts as Go/Python/Node/Rust — language-neutral
docs/openapi.yaml                              # emitted by the pipeline from the protos (do not hand-write)
```

Entrypoint per service: `Application.java` — for gRPC, a `main` that builds and starts the grpc-java `Server`; for HTTP, the Spring Boot `@SpringBootApplication` (`SpringApplication.run`). One service = one Maven module = one runnable artifact.

## A.3 Hexagonal package layout (the Java equivalents)

- **Layering & dependency rule:** identical to the Canon. The four package roots are `domain`, `application`, `infrastructure/{grpc|http}`, `infrastructure/repositories`. Dependencies point **inward only**.
- **`domain`** — **framework-free** POJOs/records derived from the proto resources, plus typed errors and constants. **The domain layer MUST NOT import Spring, JPA, Jakarta Persistence, grpc, or protobuf-generated types.** No `@Entity`, no `@Document`, no `@Component`, no `@Service`, no `io.grpc.*`. Domain is plain Java (records preferred). This is the single hard rule that keeps the core portable.
- **`application`** — use cases (`application/usecase`) and ports (`application/port`). Use cases depend only on `domain` + the port interfaces. **Ports = Java interfaces** (repository interfaces and the transaction-boundary interface). Adapters in `infrastructure/repositories` implement them.
- **`infrastructure/{grpc|http}`** — the driving/transport adapters: the gRPC `*ImplBase` impls (gRPC profile) OR the `@RestController`s (HTTP profile). Translate transport ⇄ domain, delegate to the use case, map errors.
- **`infrastructure/repositories`** — the persistence adapters: JPA `@Entity` / Mongo `@Document` classes **plus** Spring Data repositories, and the adapter classes implementing the domain ports by mapping domain ⇄ persistence model.
- **Dependency injection = Spring constructor injection**, assembled via `@Configuration` beans / component scanning in `infrastructure/config`. No field injection; no service locator reaching into domain/application. Domain and application classes are plain (no Spring annotations) where feasible — wiring lives at the infrastructure boundary.
- **Domain types derive from the proto** (shape-for-shape records mirroring the proto messages) — never decorate them with Spring/JPA/grpc. The proto stays the single source of truth; transport DTOs and persistence entities are **separate** classes that map to/from the domain records.

## A.4 Error model (Java mechanism for the Canon's taxonomy)

- A `DomainError` type — an unchecked exception (extend `RuntimeException`) carrying `code` (String) and `message`, with the same `Failure_Noun_Descriptor` message format and the orchestrator-assigned prefix (e.g. `IDN201`). Sentinel factories live in `domain/error` (e.g. `DomainError.notFound(...)`, `DomainError.alreadyExists(...)`). The application throws `DomainError`; it never throws raw `RuntimeException`/`IllegalArgumentException` for domain conditions.
- **gRPC mapping:** a shared `mapError(DomainError) -> io.grpc.StatusRuntimeException` (or a gRPC interceptor / `onError(...)` path). `1xx` → `INVALID_ARGUMENT`, `2xx` domain codes → the domain-specific status (`NOT_FOUND`, `ALREADY_EXISTS`, `FAILED_PRECONDITION`), `5xx` → `INTERNAL`. Every new code handled explicitly.
- **HTTP mapping:** a `@RestControllerAdvice` / `@ExceptionHandler(DomainError.class)` that translates each `code` to an HTTP status (`1xx` → 400, not-found → 404, forbidden → 403, conflict → 409, `5xx` → 500) and returns a JSON body `{ code, message }`.
- Direct `INVALID_ARGUMENT` / HTTP 400 is allowed **only** for request-field validation before delegating to the use case.

## A.4-HTTP HTTP transport rule (CRITICAL — no gRPC server in HTTP services)

For an **HTTP service** the **Spring Boot application is the ONLY entrypoint**. The authoritative `.proto` is still written WITH `google.api.http` annotations on every RPC (it is the deliverable's API contract and drives `docs/openapi.yaml`), **but at RUNTIME there is NO gRPC server**:

- **There MUST be no `io.grpc.Server`, no `ServerBuilder`, no `BindableService`, no `*ImplBase`, and no `infrastructure/grpc/` package.**
- Handlers are `@RestController` methods, **1:1 with the RPCs**, mounted on the method + path declared by each RPC's `google.api.http` annotation.
- Request/response messages are **POJOs/records or Jackson DTOs** (`@RequestBody`, `@PathVariable`, `@RequestParam` → `ResponseEntity`/`@ResponseBody`), not grpc-generated message types hand-marshalled through a gRPC server.
- The `protobuf-maven-plugin` runs (at most) for *message* types if desired, but **never** generates or wires a gRPC server. The recommended lightest path is plain serde/Jackson DTOs.

## A.5 Persistence — MongoDB (default; Spring Data MongoDB)

- **Driver:** `spring-boot-starter-data-mongodb`. For each owned resource define a `@Document` class in `infrastructure/repositories` (NOT in domain) and a Spring Data `interface FooMongoRepository extends MongoRepository<FooDocument, Long>`.
- **Repository adapter:** `infrastructure/repositories/MongoFooRepository implements ports.FooRepository`, mapping domain record ⇄ `@Document` on the way in/out.
- **IDs:** the `system_counters` pattern — a dedicated collection, `findAndModify` with `$inc`, `returnNew(true)`, `upsert(true)`, seeded at a reserved start value, stored as BSON `Long` and mapped to the proto `uint64 identifier`. (For SQL, IDs come from a JPA autoincrement PK instead — §A.13.)
- **Soft-delete:** a nullable `deleteTime` field; deletes set it, reads filter it out.
- **Config:** `SPRING_DATA_MONGODB_URI` / `spring.data.mongodb.uri`. **No SQL/JPA deps when store is mongodb.**

## A.6 Transaction boundary (port + adapter)

- A `TransactionManager` port interface with a `<T> T withTransaction(Supplier<T>)` (or `void withTransaction(Runnable)`) method.
- Mongo adapter implements it over a `MongoTransactionManager` / `@Transactional` session-bound context; SQL adapter implements it over Spring `@Transactional` / `PlatformTransactionManager`. A service needing no transactional boundary omits the dependency.

## A.7 Logging

- One logger mechanism: **SLF4J** (`org.slf4j.Logger` via `LoggerFactory.getLogger(...)`), the Spring Boot default Logback backend. Structured format `"<context>: key=value"`. **Forbidden:** `System.out.println` / `System.err.println` / `printStackTrace()` scattered in layers, or any ad-hoc logger. Everything goes through SLF4J.

## A.8 Testing (Java mechanism for the Canon's philosophy)

- **JUnit 5** (`spring-boot-starter-test`, which brings JUnit Jupiter + Mockito + AssertJ). Application unit tests run via `mvn -B test`. The application layer is tested against its ports using Mockito mocks of the port interfaces; Mongo/JPA are **not** mocked in application tests — the application layer only sees ports.
- Coverage floor (Canon §8): every use case has a success test + ≥1 error scenario; every error sentinel asserted by ≥1 test. Test names `<operation>_<scenario>`.
- **`mvn -B package` is the hard build gate** — it must compile + assemble with zero errors (this includes the `protobuf-maven-plugin` codegen for gRPC services). `mvn -B test` runs the unit tests.

## A.9 Java-specific verifier gates

In addition to the Canon-level gates (`buf lint`, AIP, OpenAPI emission):

- **`mvn -B package` passes** with zero errors (the primary build gate). For gRPC services this includes `protobuf-maven-plugin` codegen against the canonical proto.
- **`mvn -B test`** passes (when tests are emitted).
- **Layer-import discipline** — encode the dependency rule via package boundaries: `domain` imports only `java.*` + other `domain` (NO Spring/JPA/grpc/protobuf); `application` imports only `domain` + its ports; `infrastructure/{grpc|http}` never imports `repositories` directly. A violation is caught in the self-audit.
- **No hand-edited generated protobuf/grpc code** (it lives in `target/generated-sources/`, never in the tree, never committed).
- **HTTP services:** zero `io.grpc.Server` / `ServerBuilder` / `BindableService` / `infrastructure/grpc/` (A.4-HTTP).

## A.10 What stays a hole / Cleanliness

- **PostgreSQL / MySQL/MariaDB — FILLED (see §A.13).** When `store: postgres` or `store: mysql` the generator emits the Spring Data JPA layer described in §A.13.
- **NATS and any non-gRPC/non-HTTP transport** remain holes. If a task or boundary spec requires them, **stop and write a blocker note** — do not improvise.
- **Cleanliness (MUST):** never ship `target/`, `.class`, `.jar`, or a vendored `.m2`/`/usr/local/m2` copy in the deliverable. No platform-proto leak beyond the import closure actually used. Idiomatic Java naming (PascalCase types, camelCase members, package-per-layer). Add a `.gitignore` ignoring `target/` if one is warranted.

## A.11 Build-cost note (containerised gate)

Compiling a Spring Boot + grpc-java service is heavier than Go/Node. The generation container runs with a memory limit and a build timeout. To stay within them: keep the dependency set minimal (only A.1.1's table), lean on the pre-warmed `/usr/local/m2` to absorb downloads, build with `mvn -B -o` (offline) where the warmed repo is complete, and avoid heavy optional starters (no `actuator`/`webflux`/extra autoconfig beyond what the service needs).

## A.12 Auth / Validation

JWT is the only request-auth scheme generated in v1 (besides `none`). When the generation prompt's "Auth / Validation: JWT" section is present, implement bearer JWT validation with **`spring-boot-starter-oauth2-resource-server`** (preferred) or **`io.jsonwebtoken:jjwt-*`**:

- Read the token from `Authorization: Bearer <token>`; read the secret/public key, issuer, audience and required claims from the environment / `application.yml` (`JWT_SECRET`, `JWT_ISSUER`, `JWT_AUDIENCE`), NEVER hardcoded.
- **HTTP:** wire as a Spring Security **filter chain** (`SecurityFilterChain` bean) or an `OncePerRequestFilter`; expose the authenticated identity (e.g. `sub`) to handlers (request attribute / `SecurityContext`).
- **gRPC:** wire as a grpc-java **`ServerInterceptor`** that validates the token from the call metadata and places the identity in the gRPC `Context`.
- Reject `alg=none` and any algorithm outside the configured family (HS* symmetric; RS*/ES*/EdDSA asymmetric public key). On failure return a typed error mapped to HTTP 401 / gRPC `UNAUTHENTICATED`, without leaking the reason or the token.
- The validation must compile under `mvn -B package` and be covered by a `mvn -B test`.

A detected scheme other than JWT (OAuth2 authorization-code / session cookie / API key / Basic) is a hole in v1: do NOT improvise it. Generate without an auth layer and add a TODO — or re-run with `target_auth_scheme = AUTH_SCHEME_JWT`.

## A.13 Infrastructure — SQL adapters via Spring Data JPA / Hibernate (`store: postgres` | `store: mysql`)

When the boundary spec carries `store: postgres` or `store: mysql` (and the generation prompt's "Persistence: … (JPA)" section is present), generate a **Spring Data JPA / Hibernate** persistence layer **instead of** §A.5–A.6's MongoDB adapters. This is the Java homologue of the Go-GORM, Python-SQLAlchemy, Node-Prisma and Rust-SeaORM SQL cells: **the SAME entity set serves PostgreSQL AND MySQL/MariaDB** — only the JDBC driver dependency + the `DATABASE_URL` scheme differ (`jdbc:postgresql://…` vs `jdbc:mariadb://…`), Hibernate **auto-detects the dialect**. Domain types remain framework-free records (Canon §5.1, A.3), never JPA entities — the `@Entity` classes are **separate** structs.

- **Driver dependency.** Add `spring-boot-starter-data-jpa` PLUS exactly one JDBC driver: `org.postgresql:postgresql` for `store: postgres`, `org.mariadb.jdbc:mariadb-java-client` for `store: mysql` (it speaks both MySQL and MariaDB). Do NOT add a Mongo dep on a SQL store; do NOT add both drivers.
- **Entities live in infrastructure (NOT domain).** For each owned resource define a `@Entity` class in `infrastructure/repositories` (`@Table(name="…")`, snake_case columns via `@Column(name="…")`), the PK a `@Id @GeneratedValue(strategy = GenerationType.IDENTITY) Long`. Domain records are NEVER decorated with JPA annotations.
- **Repositories.** One Spring Data `interface Foo JpaRepository extends JpaRepository<FooEntity, Long>` per owned resource, plus a `JpaFooRepository implements ports.FooRepository` adapter mapping domain ⇄ entity. Only the implementation changes; application/handlers are unchanged.
- **Identifier generation (Canon §5.3).** Autoincrement primary key by the database (`@GeneratedValue(IDENTITY)`) — never an emulated `system_counters` table. Map the proto `uint64 identifier` to/from the entity's `Long` without losing precision.
- **Schema.** `spring.jpa.hibernate.ddl-auto=update` derives the schema from the entities on startup (themselves derived from the boundary spec's `owned_resources`). FK columns/indexes come from `cross_service_fks` (FK columns/indexes only — never a hard cross-service FK constraint, per the data-ownership boundary). Do NOT hand-write `*.sql` migrations in v1.
- **Soft-delete.** A nullable `@Column delete_time` (`Instant`/`LocalDateTime`); deletes set it (update), reads filter `delete_time IS NULL`, matching the Mongo path's semantics.
- **Transaction boundary.** Spring `@Transactional` / a `PlatformTransactionManager`-backed `TransactionManager` port adapter; a service needing no boundary omits it.
- **Config.** Read from `application.yml` / environment: `DATABASE_URL` (the full JDBC URL — `jdbc:postgresql://…` or `jdbc:mariadb://…`) and/or `SPRING_DATASOURCE_URL` / `SPRING_DATASOURCE_USERNAME` / `SPRING_DATASOURCE_PASSWORD`. **Never** hardcode a password; emit zero `MONGO_*`/`spring.data.mongodb.*` keys. **No Mongo deps when SQL and vice-versa.** The assembler ships a per-service `application.yml`/`.env.example` with these keys.
- **Testing.** Application tests still mock the port interfaces (no DB). Repository round-trips may be exercised with an in-memory/container-backed DB; `mvn -B package` + `mvn -B test` is the gate (A.8/A.9).
