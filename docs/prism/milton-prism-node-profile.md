# Milton Prism — Node (TypeScript) Language Profile

Peer document to `milton-prism-go-profile.md` and `milton-prism-python-profile.md`. This is the canonical **TypeScript + gRPC** mechanism reference for all Node services generated by Milton Prism. It composes with the **Architecture Canon** (`milton-prism-architecture-canon.md`) exactly as the Go and Python Profiles do: the Canon's principles, the proto/AIP contract, the error taxonomy, the dependency rule, and the testing philosophy all still apply unchanged. This document fills only the Node/TypeScript *mechanisms*.

> **Scope note:** this Profile defines the *output* language for generated microservices. It is independent of the analysis engine's input languages. **Persistence axis (v1).** MongoDB is filled from real patterns (the native `mongodb` driver). **SQL is now a generated cell via the Prisma ORM** — branch on the `store:` field of the boundary spec (and the generation prompt's "Persistence" section): `store: mongodb` → §A.5/§A.6's `mongodb`-driver adapters; `store: postgres` OR `store: mysql` → §A.12's Prisma adapters (ONE `schema.prisma` + `@prisma/client` + repos serve **PostgreSQL** and **MySQL/MariaDB**, the datasource `provider` + `DATABASE_URL` selected by store, schema by Prisma Migrate / `db push`). **NATS remains a hole** — if requested, stop and write a blocker note rather than improvising.

> **No reference implementation in the repo.** Unlike the Python profile (which ships a `python/` skeleton with an `identity` service to copy), the Node deliverable is generated **entirely from this document plus the proto contract**. There is no pre-existing `node/` tree to read. Follow the patterns in this document literally; they ARE the reference.

---

## A.1 Stack (fixed)

- **Language:** TypeScript 5.x, compiled with `tsc` to CommonJS (`module: commonjs`, `target: es2022`), `strict: true`.
- **Runtime:** Node.js 20+.
- **gRPC:** `@grpc/grpc-js` (pure-JS gRPC server and inter-service clients) — gRPC is the primary transport per the Canon.
- **Proto codegen (PRIMARY, npm-only — no protoc required):** `@grpc/proto-loader`. Load the `.proto` at runtime with `protoLoader.loadSync(protoPath, {keepCase:false, longs:String, enums:String, defaults:true, oneofs:true})` + `grpc.loadPackageDefinition(...)` to get the service definition and message constructors. Generate the TypeScript **type declarations** with `proto-loader-gen-types` (ships in the `@grpc/proto-loader` package; run via `npx proto-loader-gen-types --grpcLib=@grpc/grpc-js --outDir=gen/ <proto>`), emitting `.d.ts` interfaces for messages and a typed `ProtoGrpcType` for the service. This path needs ONLY npm packages — it is the default because the agent image is not guaranteed to carry `protoc`.
- **Proto codegen (ALTERNATIVE):** `ts-proto` via `protoc`/`buf` with `protoc-gen-ts_proto` — only if `protoc` is present in the environment (`command -v protoc`). If `protoc` is missing, use the primary `@grpc/proto-loader` path; do NOT fail the build trying to install a protoc binary.
- The generated stubs / type declarations live under the source root's `gen/` and are **NEVER edited by hand**.
- **Mongo driver:** the official `mongodb` package (`MongoClient`, `Db`, `Collection`) — async/await end to end.
- **Async:** the service is async end to end (`async`/`await`); never block the event loop with sync I/O.
- **Dependency management:** a single root `package.json` (workspace root) with pinned versions and a committed `package-lock.json`; `tsconfig.json` at the root with per-service `tsconfig` extending it only if needed.
- **Dependency hygiene:** `package.json` lists ONLY the packages the code imports — no unused `dependencies` or `devDependencies`; keep the set minimal.
- **IDs:** the `mongodb` driver's `Long` (BSON) for the proto `uint64 identifier` field; never coerce to JS `number` (precision loss above 2^53).

## A.2 Repository layout (mirrors the Go/Python monorepo conceptually)

The agent writes everything under the source root **`node/`** (the assembler renames `node/` → `core/` in the final deliverable, exactly as Python renames `python/` → `core/`). Protos and docs are siblings of the source root.

```
node/                                       # TypeScript monorepo source root (→ core/ in deliverable)
├── package.json                            # workspace root: deps, scripts (build/test/lint)
├── tsconfig.json                           # strict TS config, rootDir: ., outDir: dist
├── gen/<module>/...                        # ts-proto generated stubs — NEVER edited by hand
├── shared/                                 # cross-cutting (mirrors core/shared in Go, shared/ in Python)
│   ├── errors/                             # DomainError class + gRPC status mapping
│   ├── mongo/                              # MongoClient builder + lifecycle (connect/close)
│   ├── grpc-clients/                       # inter-service gRPC client builders
│   ├── interceptors/                       # logging, ctx-id, error-recovery (gRPC server interceptors)
│   ├── logging/                            # the only logger (mirrors applog)
│   ├── ids/                                # system_counters id generator
│   └── config/                             # typed config loading from env
└── services/
    └── <service>/
        ├── domain/                         # domain types (wrap proto), typed errors, constants
        ├── ports/                          # repository + transaction-boundary interfaces (abstract)
        ├── application/                    # use cases — the only place with business logic
        ├── infrastructure/
        │   ├── grpc/                       # servicer impl: deserialize → delegate → map errors
        │   └── repositories/               # mongodb implementations of ports
        ├── wire.ts                         # single composition point
        ├── index.ts                        # gRPC server entrypoint (bootstraps the server)
        └── tests/                          # vitest/jest unit tests, mirror the layers

protobuf/proto/<module>/...                 # SAME proto contracts as Go/Python — protobuf is language-neutral
docs/openapi.yaml                           # emitted by the pipeline from the protos (do not hand-write)
```

Entrypoint per service: `services/<service>/index.ts` — bootstraps the `@grpc/grpc-js` `Server`, binds the generated service implementation, and listens on `GRPC_HOST:GRPC_PORT`.

## A.3 Hexagonal mechanisms (the TypeScript equivalents)

- **Layering & dependency rule:** identical to the Canon. `domain/` depends on nothing but the language stdlib + generated proto types from `gen/`. `application/` depends only on `domain/` + `ports/`. `infrastructure/` depends inward. **Enforced** by the import boundaries below (A.9) and by `tsc` resolving only the allowed imports.
- **Ports = TypeScript `interface`s** (structural typing), not classes. Repository interfaces and the transaction-boundary interface live in `ports/`. Adapters in `infrastructure/repositories/` implement them via `implements` (or structural match — both are fine in TS).
- **Dependency injection = constructor injection**, assembled only in `wire.ts`. No global singletons, no service locator, no decorator-based DI container in the domain/application layers. This is the TS analogue of the Go `wire.go` single-composition-point rule.
- **Domain types** wrap the generated proto messages (a thin `type` alias or `interface` re-export from `gen/`) — **no parallel models** that duplicate a proto message. The proto stays the single source of truth.

## A.4 Error model (TypeScript mechanism for the Canon's taxonomy)

- A `DomainError` class (`extends Error`) carrying `code: string` and `message: string`, with the same `Failure_Noun_Descriptor` message format and the orchestrator-assigned prefix (e.g. `IDN201`). Sentinel factory functions / constants are defined in `domain/errors.ts`.
- Application throws `DomainError`. Handlers catch and **map** to a gRPC status via a shared `mapError(err): ServiceError` helper in `shared/errors` (the analogue of `coreerror` + `mapError`). The mapper returns a `{ code: status.X, message }` object passed to the gRPC callback. Direct gRPC `INVALID_ARGUMENT` is allowed **only** for request-field validation before delegating to the application.
- The mapping convention: `1xx` → `INVALID_ARGUMENT`, `2xx` domain codes → the domain-specific status (e.g. `NOT_FOUND`, `ALREADY_EXISTS`, `FAILED_PRECONDITION`), `5xx` → `INTERNAL`. Every new code is handled explicitly in the mapper (no silent default to INTERNAL for known 2xx codes).

## A.5 Transaction boundary (port + mongodb adapter)

- A `TransactionManager` interface with `withTransaction<T>(fn: (session: ClientSession) => Promise<T>): Promise<T>` — `fn` receives the session-bound context. Mirrors the Go closure mechanism, Promise-flavored.
- The mongodb adapter implements it over `client.startSession()` + `session.withTransaction(...)`. A null/undefined manager degrades to running `fn` with no session, same as Go's nil manager.

## A.6 ID generation (Mongo mechanism)

- Same `system_counters` pattern as the Go/Python Profiles: a dedicated collection, `findOneAndUpdate` with `$inc`, `returnDocument: 'after'`, `upsert: true`, seeded at a reserved start value, retry on duplicate-key. Produces conflict-free monotonic identifiers stored as BSON `Long` and mapped to the proto `uint64 identifier`.

## A.7 Logging

- One logger module in `shared/logging` (the `applog` analogue) exposing `infof/warnf/errorf`. Structured format `"<context>: key=value"`. **Forbidden:** bare `console.log` / `console.error` scattered in layers, or any ad-hoc logger. Everything goes through the shared logger.

## A.8 Testing (TypeScript mechanism for the Canon's philosophy)

- **vitest** (preferred; `jest` acceptable) for application unit tests. Application is tested against its ports using plain object/`vi.fn()` mocks; Mongo is **not** mocked in application tests — the application layer only sees ports. Repository integration tests (against a real MongoDB) are optional in v1 and may be omitted if no MongoDB is reachable in the gate environment.
- Coverage floor (Canon §8): every exported application use case has a success test + ≥1 error scenario; every error sentinel asserted by ≥1 test. Test names `<operation> <scenario>`.
- **`tsc --noEmit` (or `npm run build`) is the hard build gate** — it must pass with `strict: true` and zero errors. This is the equivalent of `go build`.

## A.9 Node-specific verifier gates

In addition to the Canon-level gates (`buf lint`, AIP, OpenAPI emission):

- **`tsc` / `npm run build` passes** with `strict: true` (the primary build gate — equivalent to `go build`).
- **ESLint** (if a config is emitted) passes; lint is best-effort and must not block if no config ships.
- **Layer-import discipline** — encode the dependency rule via directory boundaries: `domain/` imports only `gen/` + stdlib; `application/` imports only `domain/` + `ports/`; `infrastructure/grpc/` never imports `repositories/` directly. A violation is caught in the self-audit (A.8 / generator prompt §5).
- **vitest** passes (when tests are emitted).
- No hand-edited files under `gen/`.

## A.10 What stays a hole

NATS / messaging. PostgreSQL and MariaDB are NO LONGER holes — they are generated via Prisma (§A.12). If a task or boundary spec requires NATS, **stop and write a blocker note** — do not improvise an adapter.

## A.11 Auth / Validation

JWT is the only request-auth scheme generated in v1 (besides `none`). When the
generation prompt's "Auth / Validation: JWT" section is present, implement bearer
JWT validation with **`jsonwebtoken`** (or `jose`):

- Read the token from `Authorization: Bearer <token>`; read the secret/public key,
  issuer, audience and required claims from `process.env`, NEVER hardcoded. Add the
  variables to `.env.example`.
- Wire validation as a Fastify `preHandler` hook / plugin on the protected routes
  (HTTP), or a gRPC server interceptor before each handler (gRPC). Expose the
  authenticated identity (e.g. `sub`) to handlers via the request.
- Reject `alg=none` and any algorithm outside the configured family (HS* symmetric;
  RS*/ES*/EdDSA asymmetric public key). On failure return a typed error mapped to
  HTTP 401 / gRPC `UNAUTHENTICATED`, without leaking the reason or the token.
- The validation must pass `tsc --noEmit` and be covered by a unit test.

A detected scheme other than JWT (OAuth2 / session cookie / API key / Basic) is a
hole in v1: do NOT improvise it. Generate without an auth layer and add a TODO note
— or re-run with `target_auth_scheme = AUTH_SCHEME_JWT`.

## A.12 Infrastructure — SQL adapters via Prisma (`store: postgres` | `store: mysql`)

When the boundary spec carries `store: postgres` or `store: mysql` (and the generation prompt's "Persistence: … (Prisma ORM)" section is present), generate a **Prisma** persistence layer **instead of** §A.5/§A.6's `mongodb`-driver adapters. **One `schema.prisma` + one generated `@prisma/client` + one set of repos serve PostgreSQL AND MySQL/MariaDB** — only the datasource `provider` and the `DATABASE_URL` scheme differ, Prisma handles the dialect. This is the exact Node homologue of the Go Profile's §7P (GORM) and the Python Profile's §A.12 (SQLAlchemy). Domain types remain the proto-derived TypeScript types/interfaces (Canon §5.1), never ORM entities — the Prisma models are declared **separately** in `schema.prisma`. Node + MongoDB is unaffected: it keeps the native `mongodb` driver (§A.5/§A.6), never Prisma.

- **ORM & provider.** Use **Prisma** (`@prisma/client` runtime dependency, `prisma` devDependency). Define ONE `schema.prisma` with `generator client { provider = "prisma-client-js" }` and `datasource db { provider = "<provider>"; url = env("DATABASE_URL") }` where `<provider>` is `postgresql` for `store: postgres` and `mysql` for `store: mysql`. Do NOT use a raw SQL driver (`pg`/`mysql2`) or another ORM.
- **Models live in `schema.prisma`; the client is infrastructure (NOT domain).** Declare a Prisma `model` per owned resource in `schema.prisma`. The PK is `id BigInt @id @default(autoincrement())` (the DB autoincrements; map the proto `uint64 identifier` to/from `BigInt`, never a JS `number`); add a nullable `deleteTime DateTime? @map("delete_time")` for soft-delete. Snake_case table/column names via `@@map`/`@map`. Domain (proto-derived) types are NEVER decorated with Prisma types. The `schema.prisma` + a `PrismaClient` wrapper live under `services/<svc>/infrastructure` (or `shared/prisma`), NEVER in domain.
- **Repositories.** One per owned resource: `services/<svc>/infrastructure/repositories/prisma-<resource>-repository.ts`, implementing the **same** `<Resource>Repository` port interface the `mongodb` repo would (`implements <Resource>Repository`). Only the implementation changes; application/handlers are unchanged. Each method **maps domain↔Prisma-model** on the way in/out, using an injected `PrismaClient`.
- **Client builder.** `shared/prisma/client.ts` builds the `PrismaClient` once (a module singleton — the Mongo-client homologue), reads `DATABASE_URL` from config/env, configures the pool via the connection-string parameters, and `$connect()`s on startup (fail-fast). Wire it in `wire.ts` where the Mongo client was wired.
- **Transaction boundary.** A `withTransaction<T>(fn)` over `prisma.$transaction(async (tx) => …)` (a `Prisma.TransactionClient`-scoped session), null-safe, mirroring the Mongo transaction abstraction so service-layer boundaries are unchanged.
- **Schema via Prisma Migrate / db push.** The schema is derived from `schema.prisma`: apply it with `prisma migrate deploy` (or `prisma db push`) — do NOT hand-write SQL migrations and do NOT emulate a `system_counters` collection. `npx prisma generate` produces the typed client (run it as a `postinstall`/build step). Model FK columns/relations come from the `cross_service_fks` in the boundary spec (FK columns/indexes only, never a hard cross-service FK constraint).
- **Config.** Read connection settings from `.env`/environment: `DATABASE_URL` (the full Prisma URL — PostgreSQL or MySQL form) and/or `DB_HOST`/`DB_PORT`/`DB_USER`/`DB_PASSWORD`/`DB_NAME`. **Never** hardcode a password; emit zero `MONGO_*` variables. `package.json` requires `@prisma/client` (dependencies) + `prisma` (devDependencies), NOT the `mongodb` package. The assembler ships a per-service `.env.example` with these keys.
- **Gate.** The persistence code MUST pass the build gate (`npm install` + `npx prisma generate` + `tsc --noEmit`): the generated client + repos compile and at least one repository round-trip is exercised (a Prisma-mocked or container-backed test is acceptable).
