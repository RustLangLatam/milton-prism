# Milton Prism — Ruby (grpc gem / Rails API|Sinatra) Language Profile

Peer document to `milton-prism-go-profile.md`, `milton-prism-python-profile.md`, `milton-prism-node-profile.md`, `milton-prism-rust-profile.md` and `milton-prism-java-profile.md`. This is the canonical **Ruby + Bundler + grpc gem / Rails-Sinatra** mechanism reference for all Ruby services generated by Milton Prism. It composes with the **Architecture Canon** (`milton-prism-architecture-canon.md`) exactly as the Go/Python/Node/Rust/Java Profiles do: the Canon's principles, the proto/AIP contract, the error taxonomy, the dependency rule, and the testing philosophy all still apply unchanged. This document fills only the Ruby *mechanisms*.

> **Scope note:** this Profile defines the *output* language for generated microservices. It is independent of the analysis engine's input languages. Supported stores are **MongoDB** (Mongoid, §A.5–A.6) **and SQL — PostgreSQL / MySQL/MariaDB — via ActiveRecord (§A.13)**; transports are **gRPC** (the `grpc` gem) **and HTTP** (Rails API or Sinatra). Branch on the boundary spec's `store:` field: `store: mongodb` → the Mongoid documents/repos; `store: postgres` OR `store: mysql` → §A.13's ActiveRecord models/repos. **NATS** remains a hole — if requested, stop and write a blocker note rather than improvising.

> **No reference implementation in the repo.** Unlike the Python profile (which ships a `python/` skeleton), the Ruby deliverable is generated **entirely from this document plus the proto contract**. There is no pre-existing `ruby/` tree to read. Follow the patterns in this document literally; they ARE the reference.

---

## A.1 Stack (fixed)

- **Language / runtime:** **Ruby 3.x** (the agent image ships a CRuby 3 with native-extension build tools). Frozen-string-literal magic comment encouraged. Every source file must pass `ruby -c` (syntax check) and the suite must pass `bundle exec rspec`.
- **Dependency manager:** **Bundler** — a single **`Gemfile`** at the source root (`ruby/Gemfile`) declares the gem set; a `Gemfile.lock` pins it. One service = one app under `ruby/services/<service>/`; the root Gemfile is the single resolution. **Build gate: `bundle install` (resolves from the warmed GEM_HOME) + `bundle exec rspec` + `ruby -c` on every source file.**
- **Dependency hygiene:** the `Gemfile` lists ONLY the gems the code actually requires — no unused/leftover gem; keep the set minimal.
- **Warmed gem home.** The agent image ships a pre-populated, world-readable `GEM_HOME` (a warmed `bundle install` of the A.1.1 set). The generated `Gemfile` **MUST use only gems from the warmed set** at compatible versions — adding an un-warmed gem (or a native-extension gem not pre-built) will fail/stall the offline-leaning `bundle install`. Keep the gem set minimal. **Never** vendor `vendor/bundle/` into the deliverable (see A.10 Cleanliness).
- **gRPC:** the **`grpc`** gem — a `GRPC::RpcServer` hosting `GRPC::GenericService` implementations, and stubs for inter-service calls. gRPC is the primary transport per the Canon. Stubs are generated by **`grpc_tools_ruby_protoc`** (`grpc-tools` gem) at codegen time: `*_pb.rb` message classes + `*_services_pb.rb` service stubs; service impls subclass the generated `Service` base.
- **HTTP:** **Rails API** (`rails`, API-only) or **Sinatra** — the Rack app is the only entrypoint. (See A.4-HTTP and the HTTP generation prompt.)
- **Proto codegen:** `grpc_tools_ruby_protoc` invokes the bundled `protoc` against the canonical `protobuf/proto` include root; generated Ruby lands under the service `lib/` and is **NEVER edited by hand**.
- **Mongo driver:** **Mongoid** (`mongoid` gem) — `include Mongoid::Document` model classes, repositories on top.
- **SQL:** **ActiveRecord** (`activerecord` + `pg` / `mysql2`) — `ActiveRecord::Base` models, repositories (§A.13).
- **Config:** environment variables (`MONGO_URI`, `MONGODB_DATABASE`, `DATABASE_URL`, `GRPC_HOST`, `GRPC_PORT`, `JWT_SECRET`) loaded via `dotenv` / `ENV`. Never hardcode secrets.
- **IDs:** Mongo → BSON `Long` (Ruby `Integer`) for the proto `uint64 identifier` field, never narrowed. SQL → autoincrement primary key (the default ActiveRecord `bigint` PK).

### A.1.1 Pinned gem set (must match the warmed GEM_HOME)

The generated `Gemfile` MUST use exactly these gems (kept in lockstep with the agent-image author):

| Purpose | Gem |
|---------|-----|
| gRPC core | `grpc` |
| gRPC codegen | `grpc-tools` (provides `grpc_tools_ruby_protoc`) |
| Protobuf runtime | `google-protobuf` |
| Mongo store | `mongoid` |
| SQL store (ORM) | `activerecord` |
| PostgreSQL driver | `pg` |
| MySQL/MariaDB driver | `mysql2` |
| HTTP (Rails API) | `rails` (API mode) |
| HTTP (Sinatra alt.) | `sinatra` |
| App server | `puma` |
| Auth (JWT) | `jwt` |
| Config | `dotenv` |
| Tests | `rspec` |

> **Do not introduce any gem not in this table.** If a needed gem is missing from the warmed GEM_HOME, stop and report an environment blocker rather than reaching to the network. Use the Mongo gems for `store: mongodb`, the ActiveRecord+driver gems for `store: postgres|mysql` — never both store stacks at once.

## A.2 Repository layout (mirrors the Go/Python/Node/Rust/Java monorepo conceptually)

The agent writes everything under the source root **`ruby/`** (the assembler renames `ruby/` → `core/` in the final deliverable, exactly as Node renames `node/` → `core/` and Java renames `java/` → `core/`). Protos and docs are siblings of the source root.

```
ruby/                                          # source root (→ core/ in deliverable)
├── Gemfile                                    # the gem set (A.1.1); Gemfile.lock pins it
└── services/
    └── <service>/
        ├── main.rb                            # entrypoint: build & start the GRPC::RpcServer
        ├── lib/
        │   ├── <service>/
        │   │   ├── domain/                     # framework-free POROs + typed errors
        │   │   │   ├── model/                  # plain Ruby objects wrapping the proto messages
        │   │   │   └── error/                  # DomainError + sentinels (codes off error_prefix)
        │   │   ├── application/                # use cases + ports (the only business logic)
        │   │   │   ├── port/                   # repository + transaction-boundary interfaces (duck-typed)
        │   │   │   └── usecase/                # use-case classes (raise DomainError)
        │   │   └── infrastructure/
        │   │       ├── grpc/                   # gRPC: GRPC::GenericService impls   (gRPC profile)
        │   │       │   └── interceptor/        #   GRPC::ServerInterceptor (auth, logging, ctx-id)
        │   │       ├── http/                   # HTTP: Rails/Sinatra controllers    (HTTP profile)
        │   │       ├── repositories/           # Mongoid documents / ActiveRecord models + repo adapters
        │   │       └── config/                 # wiring (composition root)
        │   └── gen/                            # generated *_pb.rb + *_services_pb.rb (codegen output)
        └── spec/                              # RSpec unit tests (bundle exec rspec)

protobuf/proto/<module>/...                    # SAME proto contracts as Go/Python/Node/Rust/Java — language-neutral
docs/openapi.yaml                              # emitted by the pipeline from the protos (do not hand-write)
```

Entrypoint per service: `main.rb` — for gRPC, builds and starts the `GRPC::RpcServer`; for HTTP, boots the Rails/Sinatra Rack app under Puma. One service = one runnable app.

## A.3 Hexagonal package layout (the Ruby equivalents)

- **Layering & dependency rule:** identical to the Canon. The four roots are `domain`, `application`, `infrastructure/{grpc|http}`, `infrastructure/repositories`. Dependencies point **inward only**.
- **`domain`** — **framework-free** POROs that wrap the proto-generated message classes, plus typed errors and constants. **The domain layer MUST NOT require Mongoid, ActiveRecord, the grpc server, or Rails.** No `include Mongoid::Document`, no `< ActiveRecord::Base`, no `GRPC::*`. Domain is plain Ruby. This is the single hard rule that keeps the core portable.
- **`application`** — use cases (`application/usecase`) and ports (`application/port`). Use cases depend only on `domain` + the port objects. **Ports = duck-typed Ruby interfaces** (a documented method set; an abstract base raising `NotImplementedError` is fine). Adapters in `infrastructure/repositories` implement them.
- **`infrastructure/{grpc|http}`** — the driving/transport adapters: the `GRPC::GenericService` impls (gRPC profile) OR the Rails/Sinatra controllers (HTTP profile). Translate transport ⇄ domain, delegate to the use case, map errors.
- **`infrastructure/repositories`** — the persistence adapters: Mongoid `Document` / ActiveRecord model classes **plus** the repository adapter classes implementing the domain ports by mapping domain ⇄ persistence model.
- **Dependency injection = constructor injection**, assembled in `infrastructure/config` (the composition root). No globals reaching into domain/application.
- **Domain types wrap the proto** (POROs mirroring the proto messages) — never make them Mongoid documents or ActiveRecord models. The proto stays the single source of truth; transport objects and persistence models are **separate** classes that map to/from the domain POROs.

## A.4 Error model (Ruby mechanism for the Canon's taxonomy)

- A `DomainError` class (`< StandardError`) carrying `code` (String) and `message`, with the same `Failure_Noun_Descriptor` message format and the orchestrator-assigned prefix (e.g. `IDN201`). Sentinel factories live in `domain/error` (e.g. `DomainError.not_found(...)`, `DomainError.already_exists(...)`). The application raises `DomainError`; it never raises a bare `RuntimeError`/`ArgumentError` for domain conditions.
- **gRPC mapping:** a shared `map_error(DomainError) -> GRPC::BadStatus` (e.g. `GRPC::NotFound`, `GRPC::AlreadyExists`, `GRPC::FailedPrecondition`, `GRPC::InvalidArgument`, `GRPC::Internal`). `1xx` → `INVALID_ARGUMENT`, `2xx` domain codes → the domain-specific status, `5xx` → `INTERNAL`. Every new code handled explicitly.
- **HTTP mapping:** a `rescue_from`/Rack mapper translating each `code` to an HTTP status (`1xx` → 400, not-found → 404, forbidden → 403, conflict → 409, `5xx` → 500) returning a JSON body `{ code, message }`.
- Direct `INVALID_ARGUMENT` / HTTP 400 is allowed **only** for request-field validation before delegating to the use case.

## A.4-HTTP HTTP transport rule (CRITICAL — no gRPC server in HTTP services)

For an **HTTP service** the **Rails/Sinatra app is the ONLY entrypoint**. The authoritative `.proto` is still written WITH `google.api.http` annotations on every RPC (it drives `docs/openapi.yaml`), **but at RUNTIME there is NO gRPC server**:

- **There MUST be no `GRPC::RpcServer`, no `GRPC::GenericService`, no `*_services_pb.rb` stub wired, and no `infrastructure/grpc/` package.**
- Handlers are Rails/Sinatra routes, **1:1 with the RPCs**, mounted on the verb + path declared by each RPC's `google.api.http` annotation.
- Request/response messages are **POROs** (the `*_pb.rb` message classes may back them), serialized as JSON — not hand-marshalled through a gRPC server.

## A.5 Persistence — MongoDB (default; Mongoid)

- **Driver:** `mongoid`. For each owned resource define a Mongoid `Document` class in `infrastructure/repositories` (NOT in domain) and a repository adapter `MongoFooRepository` implementing `ports.FooRepository`, mapping domain PORO ⇄ document.
- **IDs:** the `system_counters` pattern — a dedicated collection, an atomic `$inc` find-and-modify (`return_document: :after`, `upsert: true`), seeded at a reserved start value, stored as BSON `Long` and mapped to the proto `uint64 identifier`. (For SQL, IDs come from an ActiveRecord autoincrement PK instead — §A.13.)
- **Soft-delete:** a nullable `delete_time` field; deletes set it, reads scope it out (`where(delete_time: nil)`).
- **Config:** `MONGO_URI` / `MONGODB_DATABASE` (a `mongoid.yml` or an initializer reading `ENV`). **No SQL/ActiveRecord gems when store is mongodb.**

## A.6 Transaction boundary (port + adapter)

- A `TransactionManager` port with a `with_transaction { ... }` method.
- Mongo adapter implements it over a Mongoid session (`Mongoid::Clients.default.with_session { |s| s.with_transaction { ... } }`); SQL adapter implements it over `ActiveRecord::Base.transaction`. A service needing no boundary omits the dependency.

## A.7 Logging

- One logger mechanism: **Ruby `Logger`** (stdlib `Logger`, or the Rails logger in HTTP services). Structured format `"<context>: key=value"`. **Forbidden:** `puts` / `p` / `$stderr.puts` scattered in layers, or any ad-hoc logging. Everything goes through the configured `Logger`.

## A.8 Testing (Ruby mechanism for the Canon's philosophy)

- **RSpec** (`rspec`). Application unit tests run via `bundle exec rspec`. The application layer is tested against its ports using RSpec doubles of the port objects; Mongoid/ActiveRecord are **not** mocked in application tests — the application layer only sees ports.
- Coverage floor (Canon §8): every use case has a success test + ≥1 error scenario; every error sentinel asserted by ≥1 test. Test descriptions `<operation>_<scenario>`.
- **`bundle exec rspec` is the hard build gate** — it must pass with zero failures. Every source file must also pass **`ruby -c`** (syntax check), which includes the generated `*_pb.rb`/`*_services_pb.rb`.

## A.9 Ruby-specific verifier gates

In addition to the Canon-level gates (`buf lint`, AIP, OpenAPI emission):

- **`bundle exec rspec` passes** with zero failures (the primary gate), and **`ruby -c`** passes on every `.rb` file. For gRPC services the gate includes the `grpc_tools_ruby_protoc` codegen producing `*_pb.rb` + `*_services_pb.rb` that load cleanly.
- **Layer-import discipline** — encode the dependency rule via `require` boundaries: `domain` requires only stdlib + the proto message classes (NO Mongoid/ActiveRecord/grpc server); `application` requires only `domain` + its ports; `infrastructure/{grpc|http}` never requires `repositories` directly. A violation is caught in the self-audit.
- **No hand-edited generated protobuf/grpc code** (`*_pb.rb`/`*_services_pb.rb` are codegen output).
- **HTTP services:** zero `GRPC::RpcServer` / `GRPC::GenericService` / `infrastructure/grpc/` (A.4-HTTP).

## A.10 What stays a hole / Cleanliness

- **PostgreSQL / MySQL/MariaDB — FILLED (see §A.13).** When `store: postgres` or `store: mysql` the generator emits the ActiveRecord layer described in §A.13.
- **NATS and any non-gRPC/non-HTTP transport** remain holes. If a task or boundary spec requires them, **stop and write a blocker note** — do not improvise.
- **Cleanliness (MUST):** never ship `vendor/bundle/`, `.bundle/`, `tmp/`, `*.gem`, or `coverage/` in the deliverable. No platform-proto leak beyond the import closure actually used. Idiomatic Ruby naming (CamelCase classes/modules, snake_case methods/files). Add a `.gitignore` ignoring `vendor/bundle/`, `tmp/`, `coverage/` if warranted.

## A.11 Build-cost note (containerised gate)

Ruby does NOT compile heavy native code at the application layer — the native-extension gems (`grpc`, `pg`, `mysql2`, `bson`) are pre-built into the warmed GEM_HOME. To stay within the container budget: keep the gem set minimal (only A.1.1's table), lean on the pre-warmed GEM_HOME to absorb installs (`bundle install` should resolve offline), and avoid heavy optional gems beyond what the service needs. Ruby runs on the default resource tier.

## A.12 Auth / Validation

JWT is the only request-auth scheme generated in v1 (besides `none`). When the generation prompt's "Auth / Validation: JWT" section is present, implement bearer JWT validation with the **`jwt`** gem:

- Read the token from `Authorization: Bearer <token>`; read the secret/public key, issuer, audience and required claims from `ENV` (`JWT_SECRET`, `JWT_ISSUER`, `JWT_AUDIENCE`), NEVER hardcoded.
- **HTTP:** wire as a Rack middleware or a Rails `before_action`; expose the authenticated identity (e.g. `sub`) to handlers.
- **gRPC:** wire as a **`GRPC::ServerInterceptor`** that validates the token from the call metadata and places the identity in the active-call context.
- Reject `alg=none` and any algorithm outside the configured family (HS* symmetric; RS*/ES*/EdDSA asymmetric public key). On failure raise a typed error mapped to HTTP 401 / gRPC `UNAUTHENTICATED`, without leaking the reason or the token.
- The validation must load under `ruby -c` and be covered by an RSpec test.

A detected scheme other than JWT (OAuth2 authorization-code / session cookie / API key / Basic) is a hole in v1: do NOT improvise it. Generate without an auth layer and add a TODO — or re-run with `target_auth_scheme = AUTH_SCHEME_JWT`.

## A.13 Infrastructure — SQL adapters via ActiveRecord (`store: postgres` | `store: mysql`)

When the boundary spec carries `store: postgres` or `store: mysql` (and the generation prompt's "Persistence: … (ActiveRecord)" section is present), generate an **ActiveRecord** persistence layer **instead of** §A.5–A.6's Mongoid adapters. This is the Ruby homologue of the Go-GORM, Python-SQLAlchemy, Node-Prisma, Rust-SeaORM and Java-JPA SQL cells: **the SAME model set serves PostgreSQL AND MySQL/MariaDB** — only the driver gem + the `DATABASE_URL` scheme differ (`postgres://…` vs `mysql2://…`). Domain types remain framework-free POROs (Canon §5.1, A.3), never ActiveRecord models — the model classes are **separate**.

- **Driver gem.** Add `activerecord` PLUS exactly one driver: `pg` for `store: postgres`, `mysql2` for `store: mysql` (it speaks both MySQL and MariaDB). Do NOT add `mongoid` on a SQL store; do NOT add both drivers.
- **Models live in infrastructure (NOT domain).** For each owned resource define an `ActiveRecord::Base` model in `infrastructure/repositories` (snake_case table/columns), the PK the default `bigint` autoincrement. Domain POROs are NEVER ActiveRecord models.
- **Repositories.** A `JpaFooRepository`-equivalent `ArFooRepository` adapter per owned resource implementing `ports.FooRepository` mapping domain ⇄ model. Only the implementation changes; application/handlers are unchanged.
- **Identifier generation (Canon §5.3).** Autoincrement primary key by the database — never an emulated `system_counters` table. Map the proto `uint64 identifier` to/from the model's integer id without losing precision.
- **Schema.** ActiveRecord migrations (or `schema.rb`) derive the schema from the models (themselves derived from the boundary spec's `owned_resources`). FK columns/indexes come from `cross_service_fks` (FK columns/indexes only — never a hard cross-service FK constraint, per the data-ownership boundary).
- **Soft-delete.** A nullable `delete_time` column; deletes set it (update), reads scope `where(delete_time: nil)`, matching the Mongo path's semantics.
- **Transaction boundary.** `ActiveRecord::Base.transaction`-backed `TransactionManager` port adapter; a service needing no boundary omits it.
- **Config.** Read from `ENV`: `DATABASE_URL` (the full URL — `postgres://…` or `mysql2://…`). **Never** hardcode a password; emit zero `MONGO_*` keys. **No Mongo gems when SQL and vice-versa.** The assembler ships a per-service `.env.example` with these keys.
- **Testing.** Application tests still mock the port objects (no DB). Repository round-trips may be exercised with sqlite/container-backed specs; `bundle exec rspec` + `ruby -c` is the gate (A.8/A.9).
