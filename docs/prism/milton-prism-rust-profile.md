# Milton Prism — Rust (Tonic) Language Profile

Peer document to `milton-prism-go-profile.md`, `milton-prism-python-profile.md`, and `milton-prism-node-profile.md`. This is the canonical **Rust + Tonic + gRPC** mechanism reference for all Rust services generated by Milton Prism. It composes with the **Architecture Canon** (`milton-prism-architecture-canon.md`) exactly as the Go/Python/Node Profiles do: the Canon's principles, the proto/AIP contract, the error taxonomy, the dependency rule, and the testing philosophy all still apply unchanged. This document fills only the Rust *mechanisms*.

> **Scope note:** this Profile defines the *output* language for generated microservices. It is independent of the analysis engine's input languages. Supported stores are **MongoDB** (native `mongodb` crate, §A.1–A.8) **and SQL — PostgreSQL / MySQL/MariaDB — via the SeaORM ORM (§A.13)**; transports are **gRPC** and **HTTP** (axum-native). Branch on the boundary spec's `store:` field: `store: mongodb` → the `mongodb`-crate adapters; `store: postgres` OR `store: mysql` → §A.13's SeaORM adapters. **NATS** remains a hole — if requested, stop and write a blocker note rather than improvising.

> **No reference implementation in the repo.** Unlike the Python profile (which ships a `python/` skeleton with an `identity` service to copy), the Rust deliverable is generated **entirely from this document plus the proto contract**. There is no pre-existing `rust/` tree to read. Follow the patterns in this document literally; they ARE the reference.

---

## A.1 Stack (fixed)

- **Language:** Rust 2021 edition, stable toolchain. `#![deny(warnings)]` is NOT required, but the crate must build with zero errors (`cargo build`).
- **gRPC:** **Tonic** (`tonic`) — the async gRPC server and inter-service clients. gRPC is the primary transport per the Canon.
- **Proto codegen:** **Prost** (`prost`) message types generated by **`tonic-build`** invoked from a `build.rs` at compile time. `tonic-build` shells out to a `protoc` (the `protobuf` system package provides it; pin `tonic-build` features so it uses the system `protoc` — do NOT require a vendored protoc download at build time). The generated Rust is emitted into Cargo's `OUT_DIR` and pulled in with `tonic::include_proto!("<package>")` / `pub mod ... { tonic::include_proto!(...); }`. The generated code is **NEVER edited by hand** and is not committed (it lives in `target/` via `OUT_DIR`).
- **Async runtime:** **Tokio** (`tokio`, `rt-multi-thread`, `macros`). `#[tokio::main]` on the entrypoint; the service is async end to end (`async`/`await`); never block the runtime with sync I/O.
- **Mongo driver:** the official **`mongodb`** crate (`Client`, `Database`, `Collection`) — async end to end.
- **Config:** per-service `.env` loaded with **`dotenvy`** + typed deserialization via **`envy`** (or `std::env` with explicit parsing). Fields: `MONGO_URI`, `MONGO_DATABASE`, `GRPC_HOST`, `GRPC_PORT`, `JWT_SECRET`.
- **Dependency management:** a single **Cargo workspace** at the source root (`rust/Cargo.toml` with `[workspace]`); each service is a workspace member crate under `rust/services/<service>/` with its own `Cargo.toml`. Shared scaffolding is a member crate (`rust/shared/`) or modules. Pin crate versions; `Cargo.lock` is regenerated by the consumer (the deliverable ships without it).
- **IDs:** the `mongodb`/`bson` `i64` (`bson::Bson::Int64`) for the proto `uint64 identifier` field; the BSON wire type is Long. Never narrow to a type that loses precision.

## A.2 Repository layout (mirrors the Go/Python/Node monorepo conceptually)

The agent writes everything under the source root **`rust/`** (the assembler renames `rust/` → `core/` in the final deliverable, exactly as Node renames `node/` → `core/`). Protos and docs are siblings of the source root.

```
rust/                                       # Cargo workspace source root (→ core/ in deliverable)
├── Cargo.toml                              # [workspace] root: members = ["shared", "services/*"]
├── shared/                                 # cross-cutting member crate (mirrors core/shared in Go)
│   ├── Cargo.toml
│   └── src/
│       ├── lib.rs                          # re-exports the shared modules
│       ├── errors.rs                       # DomainError enum + tonic::Status mapping
│       ├── mongo.rs                        # Client builder + lifecycle (connect)
│       ├── grpc_clients.rs                 # inter-service gRPC client builders
│       ├── interceptors.rs                 # logging, ctx-id, auth (tonic interceptors)
│       ├── logging.rs                      # the only logger (mirrors applog; tracing/log)
│       ├── ids.rs                          # system_counters id generator
│       └── config.rs                       # typed config loading from env (dotenvy/envy)
└── services/
    └── <service>/
        ├── Cargo.toml                      # member crate: deps tonic, prost, tokio, mongodb, shared
        ├── build.rs                        # tonic_build::compile_protos(<canonical .proto>)
        └── src/
            ├── main.rs                     # #[tokio::main] gRPC server entrypoint
            ├── lib.rs                       # module tree (domain/application/ports/infrastructure)
            ├── pb.rs                        # tonic::include_proto!("<package>") — generated stubs
            ├── domain/                     # domain types (wrap proto), typed errors, constants
            │   └── mod.rs
            ├── ports/                       # repository + transaction-boundary traits (abstract)
            │   └── mod.rs
            ├── application/                 # use cases — the only place with business logic
            │   └── mod.rs
            ├── infrastructure/
            │   ├── mod.rs
            │   ├── grpc/                    # tonic service impl: deserialize → delegate → map errors
            │   └── repositories/            # mongodb implementations of ports
            └── wire.rs                      # single composition point

protobuf/proto/<module>/...                 # SAME proto contracts as Go/Python/Node — protobuf is language-neutral
docs/openapi.yaml                           # emitted by the pipeline from the protos (do not hand-write)
```

Entrypoint per service: `services/<service>/src/main.rs` — builds the Tonic `Server`, registers the generated service implementation, binds and serves on `GRPC_HOST:GRPC_PORT`.

## A.3 Hexagonal mechanisms (the Rust equivalents)

- **Layering & dependency rule:** identical to the Canon. `domain/` depends on nothing but `std` + generated proto types (`pb`). `application/` depends only on `domain/` + `ports/`. `infrastructure/` depends inward. **Enforced** by Rust's module visibility and the import boundaries below (A.9).
- **Ports = Rust `trait`s** (object-safe where they must be boxed; use `#[async_trait::async_trait]` for async methods). Repository traits and the transaction-boundary trait live in `ports/`. Adapters in `infrastructure/repositories/` `impl` them.
- **The repository port + adapter MUST mirror the service contract.** The repository trait exposes EXACTLY the persistence operations REQUIRED by the RPCs the authoritative `.proto` actually defines — no more. Derive the port surface from the proto's RPC set, not from a generic CRUD template: if the proto only defines `ListUser`, the `UserRepository` port has only the read/list operation `ListUser` needs (e.g. `list`), and the adapter implements only that. Do **NOT** generate speculative CRUD (`create` / `get` / `soft_delete`) that no RPC exposes — methods wired to no gRPC endpoint and exercised only by unit tests are dead-but-tested infrastructure and are a rejection trigger. A `create`/`get`/`soft_delete`/`update`/`delete` operation belongs on the port ONLY when a `Create…`/`Get…`/`Delete…`/`Update…` RPC exists in the contract.
- **Dependency injection = constructor injection**, assembled only in `wire.rs`. Inject trait objects via `Arc<dyn Trait + Send + Sync>`. No global singletons, no `lazy_static`/`once_cell` service locator in the domain/application layers. This is the Rust analogue of the Go `wire.go` single-composition-point rule.
- **Domain types** wrap the generated Prost proto messages (a thin `type` alias or newtype over a `pb::` message) — **no parallel structs** that duplicate a proto message. The proto stays the single source of truth.

## A.4 Error model (Rust mechanism for the Canon's taxonomy)

- A `DomainError` type — an `enum` (or a struct carrying `code: String, message: String`) — implementing `std::error::Error` + `Display`, with the same `Failure_Noun_Descriptor` message format and the orchestrator-assigned prefix (e.g. `IDN201`). Sentinel constructors live in `domain/errors` (e.g. `DomainError::not_found(...)`).
- Application returns `Result<T, DomainError>`. Handlers map `DomainError` → `tonic::Status` via a shared `map_error(err) -> tonic::Status` helper in `shared::errors` (the analogue of `coreerror` + `mapError`). Direct `tonic::Status::invalid_argument` is allowed **only** for request-field validation before delegating to the application.
- The mapping convention: `1xx` → `INVALID_ARGUMENT`, `2xx` domain codes → the domain-specific status (e.g. `NOT_FOUND`, `ALREADY_EXISTS`, `FAILED_PRECONDITION`), `5xx` → `INTERNAL`. Every new code is handled explicitly in the mapper (no silent default to INTERNAL for known 2xx codes).

## A.5 Transaction boundary (port + mongodb adapter)

- A `TransactionManager` trait with an async `with_transaction` method that runs a closure inside a session-bound context. Mirrors the Go closure mechanism, async-flavored.
- The mongodb adapter implements it over `client.start_session()` + `session.start_transaction()` / commit. A `None`/absent manager degrades to running the closure with no session, same as Go's nil manager.

## A.6 ID generation (Mongo mechanism)

- Same `system_counters` pattern as the Go/Python/Node Profiles: a dedicated collection, `find_one_and_update` with `$inc`, `return_document: After`, `upsert: true`, seeded at a reserved start value, retry on duplicate-key. Produces conflict-free monotonic identifiers stored as BSON `Int64`/Long and mapped to the proto `uint64 identifier`. *(For `store: postgres` / `store: mysql`, IDs come from a SeaORM autoincrement primary key instead — see §A.13.)*

## A.7 Logging

- One logger module in `shared::logging` (the `applog` analogue) exposing `info!/warn!/error!` wrappers (backed by `tracing` or `log`+`env_logger`). Structured format `"<context>: key=value"`. **Forbidden:** bare `println!` / `eprintln!` scattered in layers, or any ad-hoc logger. Everything goes through the shared logger.

## A.8 Testing (Rust mechanism for the Canon's philosophy)

- `#[cfg(test)]` modules (preferred) or a `tests/` dir for application unit tests via `cargo test`. Application is tested against its ports using hand-written mock structs implementing the port traits; Mongo is **not** mocked in application tests — the application layer only sees ports. Repository integration tests (against a real MongoDB) are optional in v1 and may be omitted if no MongoDB is reachable in the gate environment.
- Coverage floor (Canon §8): every exported application use case has a success test + ≥1 error scenario; every error sentinel asserted by ≥1 test. Test names `<operation>_<scenario>`.
- **`cargo build` is the hard build gate** — it must compile with zero errors. This is the equivalent of `go build`. `cargo test` runs the unit tests when emitted.

## A.9 Rust-specific verifier gates

In addition to the Canon-level gates (`buf lint`, AIP, OpenAPI emission):

- **`cargo build` passes** with zero errors (the primary build gate — equivalent to `go build`). `build.rs` must successfully run `tonic-build` codegen against the canonical proto.
- **`cargo clippy`** (if available) passes; clippy is best-effort and must not block if the toolchain component is absent.
- **Layer-import discipline** — encode the dependency rule via module boundaries: `domain` uses only `pb` + `std`; `application` uses only `domain` + `ports`; `infrastructure::grpc` never uses `repositories` directly. A violation is caught in the self-audit (A.8 / generator prompt §5).
- **`cargo test`** passes (when tests are emitted).
- **Contract↔implementation alignment** — the repository port + adapter expose exactly the persistence operations required by the proto's RPC set (A.3); no speculative CRUD method that no RPC reaches. A repository method with no RPC behind it (callable only from a unit test) is a gate failure.
- **No dead code / no unused deps** (A.11) — `cargo build` is warning-clean: no `unused import` / `dead_code` / `never used` warnings, no unused crate in any `Cargo.toml`, no orphan consts/macros (`IDENTIFIER_SEED`, unused log macros). The manifest lists only crates the code uses.
- No hand-edited generated Prost/Tonic code (it lives in `OUT_DIR`, never in the tree).

## A.10 What stays a hole

- **PostgreSQL / MySQL/MariaDB — FILLED (see §A.13).** No longer holes: when `store: postgres` or `store: mysql` the generator emits the SeaORM layer described in §A.13 (SeaORM entities in `infrastructure/repositories` mapping to/from domain, repos implementing the same port traits, a `Database::connect` builder selecting the sqlx driver feature by store, a SeaORM transaction manager, `sea-orm-migration`, autoincrement IDs, nullable soft-delete column, `DATABASE_URL`/`DB_*` config). One set of SeaORM entities/repos covers both engines.
- **NATS and any non-gRPC/non-HTTP transport** remain holes. If a task or boundary spec requires them, **stop and write a blocker note** — do not improvise an adapter.

## A.11 Build-cost note (containerised gate)

Compiling a Tonic/Prost service is heavier than Go/Node. The generation container runs with a memory limit and a build timeout. To stay within them: keep the dependency set minimal (only the crates listed in A.1), prefer a single workspace build, and let the pre-warmed Cargo registry/dependency cache in the agent image absorb the crate downloads and the bulk of the first compile. Do NOT add heavy optional crates (e.g. extra TLS backends, full `tracing-subscriber` feature sets) beyond what the service needs.

**No dead code / no unused dependencies.** Emit ONLY what the generated code uses. The `cargo build` gate must be warning-clean (no `unused import`, `dead_code`, `unused crate dependency`, or `never used` warnings). Concretely:
- Every crate in a `Cargo.toml` `[dependencies]`/`[build-dependencies]` is `use`d by that crate's code. Do NOT declare a dependency the service never references — e.g. if config is parsed with `std::env`/`dotenvy` and `envy` is not actually used, do NOT list `envy`. Strip any crate the chosen config/persistence path leaves unused.
- Do NOT emit consts, statics, or macros that nothing references — no orphan `pub const IDENTIFIER_SEED`, no unused `log_warn!`/logging macro left behind by boilerplate. Generate an id seed const only when the `system_counters` generator actually reads it; generate a log macro only when a layer calls it.
- This is boilerplate discipline: the generator must not stamp out scaffolding "just in case." If a piece of generated code is unreachable from the contract (no RPC, no use case, no caller), it should not exist.

## A.12 Auth / Validation

JWT is the only request-auth scheme generated in v1 (besides `none`). When the
generation prompt's "Auth / Validation: JWT" section is present, implement bearer
JWT validation with the **`jsonwebtoken`** crate:

- Read the token from `Authorization: Bearer <token>`; read the secret/public key,
  issuer, audience and required claims from the environment (`std::env::var` /
  config), NEVER hardcoded. Add the variables to `.env.example`.
- Wire validation as an axum middleware (`tower` / `axum::middleware::from_fn`) or an
  extractor on the protected `Router` (HTTP), or a tonic `Interceptor` (gRPC). Expose
  the authenticated identity (e.g. `sub`) to handlers via request extensions.
- Reject `alg=none` and any algorithm outside the configured family (HS* symmetric;
  RS*/ES*/EdDSA asymmetric public key). On failure return a typed error mapped to
  HTTP 401 / gRPC `UNAUTHENTICATED` via `IntoResponse`/`shared::errors`, without
  leaking the reason or the token. Keep the crate set minimal (A.11 build-cost note).
- The validation must compile under `cargo build` and be covered by a `cargo test`.

A detected scheme other than JWT (OAuth2 / session cookie / API key / Basic) is a
hole in v1: do NOT improvise it. Generate without an auth layer and add a TODO note
— or re-run with `target_auth_scheme = AUTH_SCHEME_JWT`.

## A.13 Infrastructure — SQL adapters via SeaORM (`store: postgres` | `store: mysql`)

When the boundary spec carries `store: postgres` or `store: mysql` (and the generation prompt's "Persistence: … (SeaORM)" section is present), generate a **SeaORM** persistence layer **instead of** §A.5–A.6's MongoDB adapters. This is the Rust homologue of the Go-GORM (§7P), Python-SQLAlchemy and Node-Prisma SQL cells: **one set of SeaORM entities/repos serves PostgreSQL AND MySQL/MariaDB** — only the sqlx driver feature in `Cargo.toml` and the `DATABASE_URL` scheme differ, SeaORM handles the dialect. Domain types remain aliases/newtypes over the prost proto messages (Canon §5.1), never ORM entities — the SeaORM entities are **separate** structs.

- **ORM & driver.** Use **SeaORM** (`sea-orm` with `runtime-tokio-rustls`) and `sea-orm-migration`, backed by sqlx with the driver feature **`sqlx-postgres`** for `store: postgres` and **`sqlx-mysql`** for `store: mysql`. Open the connection with `Database::connect(DATABASE_URL)` (async). Do NOT use raw sqlx/SQL, the `mongodb` crate, diesel, or another ORM.
- **Entities live in infrastructure (NOT domain).** For each owned resource define a SeaORM entity in `infrastructure/repositories` (`#[derive(DeriveEntityModel)]` `Model` with `#[sea_orm(...)]` column attrs, plus the generated `Entity`/`ActiveModel`). The PK is a `#[sea_orm(primary_key)]` `i64` column (auto-increment is the default for a single integer PK). Snake_case `table_name`/column names (`#[sea_orm(table_name = "…")]`). Domain (proto-alias) types are NEVER decorated with SeaORM derives.
- **Repositories.** One per owned resource: `infrastructure/repositories/seaorm_<resource>_repository.rs`, `#[async_trait] impl ports::<Resource>Repository for SeaOrm<Resource>Repository`, the **same** trait the Mongo repo would. Only the implementation changes; application/handlers are unchanged. Each method **maps domain↔SeaORM-entity** on the way in/out (`Model` → domain on read, domain → `ActiveModel` on write).
- **Connection builder.** A shared `shared::db` (e.g. `shared/seaorm.rs`) follows the Mongo-client shape: a `DatabaseConnection` built once from typed config via `Database::connect(ConnectOptions::new(database_url))`, the sqlx feature selected at compile time by store, the pool configured on `ConnectOptions` (`max_connections`/`min_connections`/`connect_timeout`/`sqlx_logging`), a fail-fast ping on connect, lifecycle held behind `Arc<DatabaseConnection>`.
- **Transaction manager.** Implements `ports::TransactionManager`'s async `with_transaction` over `db.transaction::<_, _, DbErr>(|txn| Box::pin(async move { … }))` (a `DatabaseTransaction`-scoped handle), mirroring the Mongo one so a service needing no transactional boundary omits the dependency.
- **Identifier generation (Canon §5.3).** Autoincrement primary key by the database (the single integer PK auto-increments) — never an emulated `system_counters` table. Map the proto `uint64 identifier` to/from the entity's `i64` without losing precision.
- **Schema (`sea-orm-migration`).** Provide a `Migrator` (`MigratorTrait`) whose migrations create the tables from the entities (`SchemaManager`/`create_table`, or `Schema::new(backend).create_table_from_entity(Entity)`); run `Migrator::up(&db, None).await` on startup, deriving the schema from the entities (themselves derived from the boundary spec's `owned_resources`). FK columns/indexes come from `cross_service_fks` (FK columns/indexes only — never a hard cross-service FK constraint, per the data-ownership boundary). Do NOT hand-write raw `*.sql`.
- **Soft-delete.** A nullable timestamp column (`delete_time: Option<DateTimeUtc>`); deletes set the column via an `ActiveModel` update instead of a hard `DELETE`, and reads filter `.filter(Column::DeleteTime.is_null())`, matching the Mongo path's soft-delete semantics.
- **Config.** Read connection settings from `.env`/environment (`dotenvy`/`envy`/`std::env`): `DATABASE_URL` (the full SeaORM URL — `postgres://…` or `mysql://…`) and/or `DB_HOST`/`DB_PORT`/`DB_USER`/`DB_PASSWORD`/`DB_NAME`. **Never** hardcode a password; emit zero `MONGO_*` variables. The service `Cargo.toml` requires `sea-orm` (with `runtime-tokio-rustls` + the matching sqlx feature) and `sea-orm-migration`, NOT `mongodb`/`bson`. The assembler ships a per-service `.env.example` with these keys. Keep the crate set minimal (the A.11 build-cost note still applies).
- **Testing.** Application tests still mock the port traits (no DB). Repository round-trips may be exercised with SeaORM's `MockDatabase` or a container-backed test; `cargo build` + `cargo test` is the gate (A.8/A.9).
